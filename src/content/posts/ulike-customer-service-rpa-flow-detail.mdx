很多人看到“客服售后自动化”，会先想到机器人自动聊天、AI 自动理解用户诉求、系统自动做售后决策。

但从 Ulike 这个公开案例来看，它更像是另一回事：**RPA 代替客服和售后人员，在多个后台之间反复查询、复制、判断、留言、备注、发群消息。**

<Callout title="资料边界">
  公开案例里明确提到了 Ulike 客服部门的三个应用：静默单推送、已发货仅退款、留言赠品发货信息。本文不是官方逐点击脚本，而是基于公开描述和电商客服常见流程，对 RPA 实际执行动作做的流程还原。参考来源：[影刀社区公开案例](https://www.yingdao.com/community/detaildiscuss?id=85640423-6663-4c89-9375-a50fea6421c7)。
</Callout>

## 一句话结论

<HighlightBox eyebrow="结论" title="客服售后 RPA 的价值，不是替客服思考，而是替客服完成重复后台动作">
  这三个流程的共同点是：先找到待处理订单，再补齐聊天、物流、ERP 等上下文，最后按规则完成推送、退款辅助或买家留言。真正被自动化的，是查后台、复制字段、判断状态、发送消息和回写结果。

## 先把 RPA 想成一个“后台操作员”

这类流程不要先想成 AI，也不要先想成系统架构。

更准确的理解是：RPA 像一个人坐在电脑前，按固定步骤操作后台。

```text
打开平台后台
筛选订单或售后单
复制订单号
打开 ERP 查询订单
复制物流单号
查询物流状态
回到平台处理或留言
把结果写回表格或系统
```

人工每天重复这些动作，RPA 只是把它们固化成流程。

## 一、静默单推送：RPA 自动把未付款订单推给售前客服

### 这个流程到底在做什么？

静默单推送不是自动退款，也不是自动回复用户。

它的核心是：**找到那些自己下单、还没付款、也没有明显客服跟进的订单，然后及时发到售前群。**

人工原来要不断刷新订单后台，看有没有新未付款订单，再去客服后台核实这个买家有没有咨询过。如果没有咨询，或者没人跟进，就复制订单信息发群，让售前客服去催付。

RPA 做的就是这套重复动作。

### 人工动作还原

一个售前客服原来可能这样做：

```text
1. 打开店铺订单后台
2. 筛选未付款订单
3. 按下单时间排序
4. 复制订单号、买家、商品、金额
5. 打开客服后台
6. 搜索这个买家的聊天记录
7. 判断是否有人跟进过
8. 把订单信息发到客服群
9. 标记这条订单已经提醒过
```

这些动作本身不难，但它们有两个特点：高频、怕漏。

未付款订单如果过几个小时才跟进，买家可能已经放弃了；如果同一条订单重复推送，又会打扰客服群。

### RPA 实际流程

更细一点，RPA 会先进入订单后台：

```text
打开商家后台
进入订单管理
选择订单状态：未付款
选择时间范围：最近 30 分钟或最近 2 小时
点击查询
等待订单列表加载完成
```

然后逐行读取订单：

```text
订单号
买家昵称
商品名称
SKU
订单金额
下单时间
付款状态
店铺名称
```

读到订单后，RPA 不能马上推送，还要先查这条订单有没有被处理过。

```text
如果订单号已经存在于已推送记录：
    跳过
否则：
    继续查聊天记录
```

这一步很关键。没有去重记录，机器人每运行一次，就可能把同一批未付款订单重复发到群里。

接着 RPA 打开客服后台，搜索买家聊天记录：

```text
进入客服工作台
打开聊天记录查询
输入买家昵称或订单号
点击搜索
读取最近聊天记录
判断是否有人工客服跟进
```

判断逻辑通常可以很简单：

| 聊天情况 | RPA 判断 |
| --- | --- |
| 没有聊天记录 | 可能是静默下单，推送售前跟进 |
| 只有系统自动消息 | 可能仍算静默下单，推送售前跟进 |
| 有人工客服沟通 | 说明有人跟进，可跳过或降低优先级 |
| 买家明确表示不买了 | 跳过，不推送 |
| 买家咨询过价格、赠品、优惠但未付款 | 可作为高价值跟进单推送 |

最后 RPA 拼接消息，发送到售前客服群：

```text
【静默未付款订单提醒】

平台：天猫
店铺：Ulike 官方旗舰店
买家：xxx
订单号：123456789
商品：Ulike 脱毛仪 xxx
金额：1299 元
下单时间：14:23
未付款时长：28 分钟
聊天记录：未发现人工跟进

请售前客服及时跟进。
```

发送完成后，RPA 还要回写结果：

```text
订单号：123456789
推送时间：2026-07-03 14:51
推送状态：成功
处理人：RPA
```

### 这个流程真正省掉了什么？

它省掉的不是客服销售能力，而是这些机械动作：

```text
反复刷新未付款订单
逐个复制订单信息
来回搜索聊天记录
手工判断是否跟进过
手工发群提醒
手工记录已推送
```

最终效果是：售前客服不用一直盯后台，RPA 会把需要跟进的未付款订单送到面前。

## 二、已发货仅退款：RPA 先查物流，再辅助处理售后

### 这个流程到底在做什么？

已发货仅退款，比静默单复杂得多。

因为商品已经发出去了，买家却申请“仅退款”。如果商家直接同意，可能出现货款两空：买家收到货，商家又退款。

所以人工售后或退款财务必须先核实：货到底在哪？

```text
快递是否揽收？
是否运输中？
是否已签收？
是否拒收？
是否退回仓库？
```

RPA 的作用不是简单点“同意退款”，而是先跨系统核实订单和物流状态，再按 SOP 辅助处理。

### 人工动作还原

人工处理一条已发货仅退款，通常要这样做：

```text
1. 打开平台售后后台
2. 筛选待处理的仅退款申请
3. 点进售后详情
4. 复制订单号
5. 打开 ERP 查询订单
6. 复制物流公司和物流单号
7. 查询物流轨迹
8. 判断当前物流状态
9. 回到平台售后后台
10. 同意退款、拒绝退款、留言说明或转人工
11. 记录处理结果
```

一条售后单可能要在平台后台、ERP、物流页面、记录表之间来回切换。

RPA 适合做的就是这种跨系统核实。

### RPA 实际流程

<Mermaid
  title="已发货仅退款的 RPA 判断链"
  caption="核心不是自动退款，而是先查清物流状态，再决定能否处理。"
  chart={`flowchart TD
    A[售后后台筛选已发货仅退款] --> B[读取售后单和订单号]
    B --> C[进入 ERP 查询发货信息]
    C --> D[获取物流公司和单号]
    D --> E[查询物流轨迹]
    E --> F{物流状态}
    F -->|未揽收| G[可尝试拦截或退款]
    F -->|运输中| H[备注运输中或转人工]
    F -->|已签收| I[不直接仅退款 转人工]
    F -->|拒收退回| J[等待回仓或退款处理]
    G --> K[回写处理结果]
    H --> K
    I --> K
    J --> K`}
/>

流程开始时，RPA 进入平台售后后台：

```text
打开平台商家后台
进入售后管理
售后类型选择：仅退款
订单状态选择：已发货
处理状态选择：待处理
点击查询
```

然后逐条读取售后单信息：

```text
售后单号
订单号
买家昵称
退款金额
退款原因
申请时间
商品名称
SKU
平台售后状态
```

读取后先做基础过滤：

```text
如果不是仅退款：跳过
如果不是已发货：跳过
如果已被人工处理：跳过
如果退款金额异常：转人工
如果申请原因包含高风险词：转人工
```

接着 RPA 复制订单号，进入 ERP 查询订单：

```text
打开 ERP
进入订单查询
粘贴平台订单号
点击搜索
打开订单详情
读取发货状态
读取出库时间
读取物流公司
读取物流单号
读取仓库状态
```

这一步是为了确认平台状态和 ERP 状态是否一致。

例如平台显示已发货，但 ERP 可能只是生成了物流单，快递还没有揽收；也可能 ERP 已出库，但物流轨迹还没有同步。

拿到物流单号后，RPA 继续查物流：

```text
复制物流单号
打开物流轨迹页面
粘贴单号
点击查询
等待轨迹加载
读取最新物流状态
读取最新更新时间
```

### 关键判断：不同物流状态对应不同动作

这个流程最重要的是物流状态判断。

| 物流状态 | RPA 可能动作 | 原因 |
| --- | --- | --- |
| 已发货但未揽收 | 备注可拦截，部分场景可退款 | 货可能还没真正交给快递 |
| 已揽收运输中 | 不直接退款，备注运输中或转人工 | 货还在路上，直接退款风险较高 |
| 派送中 | 转人工或等待更新 | 接近签收，不能轻易仅退款 |
| 已签收 | 不自动同意仅退款 | 需要核实是否应改为退货退款 |
| 拒收退回中 | 备注等待退回仓库 | 货还没回到商家手里 |
| 已退回仓库 | 可进入退款处理 | 货已回仓，退款风险降低 |
| 物流异常 | 转人工 | 状态不清楚，不能自动处理 |

如果符合自动处理规则，RPA 会回到售后后台执行动作。

例如：已退回仓库，可以处理退款：

```text
回到平台售后详情
点击同意退款
核对退款金额
填写处理备注
点击提交
等待平台提示成功
```

如果物流运输中，RPA 可能只做备注或留言：

```text
填写售后备注：订单已发货，物流运输中，需等待商品退回后处理退款。
点击保存备注
转入人工跟进清单
```

如果物流显示已签收，RPA 不应该直接拒绝或退款，而是把信息带给人工：

```text
标记：物流已签收
备注：买家申请仅退款，请人工核实是否改退货退款
加入人工处理表
```

### 这个流程真正省掉了什么？

它省掉的是售后人员最耗时的核实动作：

```text
复制订单号
切换 ERP
查发货状态
复制物流单号
查物流轨迹
回到平台备注
记录处理状态
```

同时它减少了漏查和误判。

售后人员不需要每一单都从头查一遍，而是先看 RPA 给出的结果：物流状态是什么、机器人做了什么、为什么转人工。

## 三、留言赠品发货信息：RPA 自动把 ERP 里的赠品物流发给买家

### 这个流程到底在做什么？

电商活动里经常有赠品，比如下单赠、评价赠、直播间赠品、活动礼包。

赠品有时不会和主商品一起发货，而是后续单独发出。买家就会问：赠品发了吗？单号多少？为什么主商品到了，赠品还没到？

人工客服要回答这个问题，就要去 ERP 查赠品发货信息，再回到平台给买家留言。

这个流程本质上是信息搬运：**ERP 里已经有赠品物流，但买家不知道，客服要把它搬到平台留言里。**

### 人工动作还原

人工客服原来可能这样做：

```text
1. 打开 ERP
2. 搜索原订单号
3. 找到赠品发货单
4. 复制赠品名称
5. 复制快递公司
6. 复制物流单号
7. 打开平台订单后台
8. 搜索买家订单
9. 进入留言页面
10. 粘贴固定话术
11. 填入赠品名称和单号
12. 点击发送
13. 回到 ERP 或表格标记已通知
```

这类工作很典型：不难，但很碎，而且数量一多就耗时间。

### RPA 实际流程

  title="赠品发货留言的执行路径"
  items={[
    {
      label: 'Step 1',
      title: '找到待通知赠品单',

    },
    {
      label: 'Step 2',
      title: '校验赠品信息',
      description: '检查赠品名称、物流公司、物流单号是否完整，避免发送空内容。',
    },
    {
      label: 'Step 3',
      title: '进入平台留言',
      description: '根据原订单号或买家昵称打开平台订单详情或客服会话。',
    },
    {
      label: 'Step 4',
      title: '发送并标记已通知',
      description: '拼接固定话术发送给买家，再把通知状态写回 ERP 或表格。',
    },
  ]}
/>

RPA 首先从 ERP 或待通知表拿到赠品单：

```text
打开 ERP
进入赠品发货查询
选择日期范围
筛选状态：已发货
筛选通知状态：未通知
点击查询
```

读取字段：

```text
原订单号
买家昵称
赠品名称
赠品数量
快递公司
物流单号
发货时间
通知状态
```

然后做完整性检查：

```text
如果物流单号为空：跳过，记录无物流单号
如果赠品名称为空：跳过，记录赠品名称为空
如果订单已经通知过：跳过，避免重复留言
如果平台订单查不到：记录异常，转人工
```

确认信息完整后，RPA 进入平台订单后台：

```text
打开平台商家后台
进入订单管理
输入原订单号
点击搜索
打开订单详情
点击联系买家或留言入口
```

接着拼接留言：

```text
亲，您的赠品已为您发出，请注意查收～

赠品：xxx
快递：xxx
单号：xxx

赠品与主商品可能分开发出，如您先收到主商品，请不用担心。
```

如果一个订单有多个赠品，RPA 也可以合并成一条留言：

```text
亲，您的活动赠品已陆续发出：

赠品1：xxx，快递：xxx，单号：xxx
赠品2：xxx，快递：xxx，单号：xxx

请注意查收。
```

发送后，RPA 要判断是否成功：

```text
消息窗口出现刚发送的内容
或页面提示发送成功
或留言记录中出现当前时间
```

成功后回写通知状态：

```text
通知状态：已通知
通知时间：2026-07-03 15:20
通知渠道：平台留言
留言结果：成功
```

失败也要记录原因：

```text
订单未找到
买家无法留言
留言按钮不可点击
物流单号为空
平台页面加载失败
```

### 这个流程真正省掉了什么？

它省掉的是“ERP 到平台”的人工搬运。

```text
查赠品单
复制赠品名称
复制快递单号
切换平台后台
搜索订单
粘贴话术
发送留言
标记已通知
```

这个流程看起来简单，但很适合 RPA，因为它的判断规则非常明确：有单号、有赠品、有对应订单，就可以留言；信息不完整，就转人工。

这三条流程可以抽象成同一个模板：

```text
1. 找到待处理数据
   未付款订单、售后单、赠品发货单

2. 补齐上下文
   聊天记录、ERP 订单、物流轨迹、赠品信息

3. 按规则判断
   是否静默、是否可退款、是否可留言

4. 执行动作
   发群消息、填写备注、同意退款、买家留言

5. 记录结果
   已推送、已处理、已通知、失败原因

6. 异常转人工
   状态冲突、信息缺失、平台限制、风险订单
```

## 为什么这些流程适合 RPA？

### 1. 规则清楚

这三个场景都有明确判断条件。

```text
未付款 + 无人工跟进 → 推送售前
已发货 + 仅退款 → 先查物流再处理
赠品已发货 + 有单号 → 给买家留言
```

规则越清楚，RPA 越容易稳定落地。

### 2. 操作重复

这些工作不是偶尔发生，而是每天都会发生。

客服和售后人员真正消耗时间的地方，往往不是“判断一次”，而是每天重复查很多次。

```text
每天查未付款订单
每天查售后物流
每天复制赠品单号
每天给买家留言
```

### 3. 系统分散

电商客服最烦的地方，是信息不在一个系统里。

```text
订单在平台后台
售后在售后后台
发货在 ERP
物流在快递系统
通知在群聊或买家留言
```

RPA 擅长的正是这种跨系统操作。

### 4. 异常可以交给人工

这些流程不需要一开始就追求 100% 自动化。

更合理的做法是：

```text
标准单自动处理
缺字段的跳过
状态冲突的转人工
风险高的转人工
处理失败的记录原因
```

这比强行全自动更稳。

## 对我自己的启发

这个案例最有价值的地方，不是 Ulike 做了多高级的系统，而是它选择的切入点非常现实。

它没有一上来做“全自动客服”，而是先找客服部门最重复、最明确、最容易验证的动作：

```text
未付款订单提醒
已发货仅退款核实
赠品物流留言
```

这也是做影刀流程时更稳的思路。

先不要问“能不能做一个完整智能客服”。

应该先问：

```text
客服每天有没有重复查后台？
有没有固定判断规则？
有没有大量复制粘贴？
有没有固定话术？
有没有需要发群或留言的动作？
有没有失败后可以转人工的边界？
```

如果这些问题答案都是“有”，那就很适合先做成 RPA。

<HighlightBox eyebrow="复盘" title="RPA 最适合先吃掉那些重复、明确、可回退的流程">
  客服售后自动化不要一开始追求替代人，而是先把人每天重复查、重复复制、重复提醒、重复记录的动作拿掉。能标准化的自动处理，不能标准化的带着原因交给人工，这样才更容易稳定落地。
</HighlightBox>
