这篇 Demo 用来检查文章里的 Markdown 表格样式：边框是否明显、表头是否清楚、长表格是否能横向滚动。

<Callout title="使用场景">
  文章里遇到对比项、规则清单、字段说明、业务看板、判断矩阵时，优先考虑 Markdown 表格；流程和因果链路继续用 Mermaid 或 DecisionFlow。
</Callout>

## 基础对比表

适合轻量对比、方案选择、优缺点说明。

| 表达方式 | 适合内容 | 不适合内容 |
| --- | --- | --- |
| 普通段落 | 观点说明、背景铺垫、结论解释 | 很多并列字段 |
| Markdown 列表 | 简短清单、步骤备注、注意事项 | 横向对比 |
| Markdown 表格 | 对比项、字段说明、判断标准 | 因果流程、复杂关系 |
| Mermaid 图 | 流程、架构、依赖关系 | 简单字段说明 |

## 业务判断表

适合把“看到什么现象 → 判断什么问题 → 下一步做什么”讲清楚。

| 现象 | 可能说明什么 | 下一步动作 |
| --- | --- | --- |
| 全店退款率都高 | 更像产品、详情页、预期管理或批次问题 | 看评价关键词、退款原因、售后反馈，再决定是否优化页面或排查批次 |
| 单店退款率偏高 | 更像店铺流量、推广人群、客服承接或链接页面问题 | 对比该店铺的推广来源、客服话术、链接详情页和活动人群 |
| 某个链接退款率偏高 | 更像链接承接问题，不一定是型号本身有问题 | 单独看链接标题、主图、SKU 描述、活动价格和退款原因 |
| 类目长期退款率高 | 更像类目天然风险或选购门槛问题 | 拆成类目专题，例如保护套机型匹配、尺寸、颜色和包装问题 |

## 字段说明表

适合给报表、流程、自动化应用做字段说明。

| 字段 | 含义 | 示例 | 备注 |
| --- | --- | --- | --- |
| 店铺 | 数据所属店铺 | BOW外设旗舰店 | 用统一店铺名称，避免同店多名 |
| 型号 | 商品型号或规格简称 | HB354、X55C、K98 | 建议统一成经营看板里的主型号 |
| 退款率 | 退款金额或退款订单占比 | 6.3% | 口径要在文章或报表中说明 |
| 风险等级 | 根据退款率、评价、库存等综合判断 | 正常 / 预警 / 高风险 | 不建议只用单一指标决定 |
| 建议动作 | 下一步人工处理方向 | 排查详情页、客服话术、批次问题 | 表格给建议，人做最终判断 |

## 写作建议

| 场景 | 推荐写法 | 原因 |
| --- | --- | --- |
| 角色分工 | 表格 | 角色、职责、边界适合横向比较 |
| 日报 vs 看板 | 表格 | 对比项明确，读者可以快速扫读 |
| 应用分级 | 表格 | 等级、特点、管理方式天然结构化 |
| 需求判断 | 表格 + 流程图 | 表格负责规则，流程图负责判断路径 |
| 因果链路 | Mermaid | 比如“歪需求 → 产出差 → 放弃使用”更像流程 |

<HighlightBox eyebrow="TABLE" title="表格适合承载结构，不适合替代所有表达">
  有横向对比、字段说明、判断标准时用表格；有先后顺序、分支判断、数据流向时用流程图。表格和图不是互相替代，而是各自负责不同阅读场景。
</HighlightBox>
