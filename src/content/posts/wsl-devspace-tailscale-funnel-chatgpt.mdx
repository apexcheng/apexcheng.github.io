之前我用的是 **WSL DevSpace + Caddy + ChmlFrp**：DevSpace 在 WSL 里监听本地端口，Caddy 做本地 HTTPS，ChmlFrp 再把 Caddy 暴露到公网。它能跑，但链路长，故障点也多。

这次改成更适合长期使用的方案：**DevSpace 和 Tailscale 都安装在 WSL 里，由 Tailscale Funnel 直接提供公网 HTTPS 地址给 ChatGPT。**

本文假设你已经在 WSL 中安装好 DevSpace，并且至少能手动启动一次。下面只讲如何固定启动方式、打开 Funnel、验收 OAuth，最后接入 ChatGPT。

<HighlightBox eyebrow="最终地址" title="给 ChatGPT 的只有 Funnel 公网地址">
  `https://devspace-wsl.your-tailnet.ts.net/mcp`
</HighlightBox>

## 最终架构与验收顺序

这篇文章真正要记住的不是命令，而是验收顺序：**本地 DevSpace 和 OAuth 先通，再开 Funnel，最后让 ChatGPT 做一次真实调用。**

<DevSpaceFunnelVerification />

普通 Tailscale 的 `100.x` 地址只给 Tailnet 内的设备访问。ChatGPT 不在你的 Tailnet 里，所以不能填 `http://100.x.x.x:7676/mcp`，必须使用 Funnel 提供的公网 HTTPS 地址。

和旧方案相比，链路从：

```text
ChatGPT -> ChmlFrp -> Caddy -> DevSpace
```

缩短为：

```text
ChatGPT -> Tailscale Funnel -> DevSpace
```

这里最重要的取舍是：**Tailscale 和 DevSpace 都放在 WSL 里。** 不要把 Windows Tailscale 当公网入口，再转发回 WSL，否则会多一层跨系统代理，排查面也会重新变复杂。

## 开始前准备

先确认下面几件事：

- WSL 已启用 `systemd`。
- DevSpace 已经安装，并且能在 WSL 中手动启动。
- 已有 Tailscale 账号，并有权限为这台设备启用 Funnel。
- 已经确定 DevSpace 只允许访问哪些项目目录。

<Callout title="公网暴露前先确认安全边界">
  Funnel 是公网入口，而 DevSpace 具备读取文件、修改项目和执行命令的能力。必须保留 OAuth，只开放必要的 `allowedRoots`，不要直接开放整个 `/home` 或根目录 `/`，也不要为了排错临时关闭认证。
</Callout>

## 开始前替换这些占位符

全文使用下面的占位符，复制命令时先替换成自己的真实值。

| 占位符 | 替换内容 |
| --- | --- |
| `YOUR_WSL_USER` | WSL 用户名 |
| `devspace-wsl` | WSL 这台 Tailscale 设备的 hostname |
| `your-tailnet.ts.net` | 你的 Tailscale Funnel 域名后缀 |
| `https://devspace-wsl.your-tailnet.ts.net` | Funnel 提供的公网 Origin |
| `/home/YOUR_WSL_USER/projects/example` | 允许 DevSpace 访问的项目目录 |
| `YOUR_DEVSPACE_START_COMMAND` | 这台机器真正能启动 DevSpace 的绝对命令 |

<Callout title="隐私提醒">
  公开笔记里建议写占位符，不要直接提交真实 tailnet 域名、私有项目路径、token 或授权文件。
</Callout>

## 第一步：先确定 DevSpace 的真实启动方式

这一步不要先照抄 systemd。先在 WSL 里手动启动一次 DevSpace，确认这台机器真正使用的启动命令。

### 情况 A：`devspace` CLI 在 PATH 里

先检查：

```bash
command -v node
command -v devspace
devspace --help
```

如果能找到 `devspace`，并且当前版本支持下面的方式启动：

```bash
devspace serve
```

那么后面的 systemd 可以直接使用它的绝对路径，例如：

```text
ExecStart=/absolute/path/to/devspace serve
```

### 情况 B：`command -v devspace` 找不到，但 DevSpace 已经能运行

这不一定是安装失败。Node 可能来自 NVM、Volta、asdf、某个工具内置环境或其他受管目录，DevSpace 也可能通过 Node 脚本启动。

这种情况不要强行把 `devspace` 加进全局 PATH，直接记录已经验证可用的绝对启动命令，例如：

```text
ExecStart=/absolute/path/to/node /absolute/path/to/start-script.mjs
```

核心原则只有一个：**systemd 使用的必须是实际能启动成功的绝对命令，不要假设每台机器都有全局 `node` 和 `devspace`。**

## 第二步：在 WSL 安装并登录 Tailscale

安装：

```bash
curl -fsSL https://tailscale.com/install.sh | sh
```

启动服务：

```bash
sudo systemctl enable --now tailscaled
```

登录，并给这台 WSL 一个稳定 hostname：

```bash
sudo tailscale up --hostname=devspace-wsl
```

如果后续希望普通用户也能管理 Funnel，可以设置 operator：

```bash
sudo tailscale set --operator=$USER
```

确认设备在线：

```bash
tailscale status
```

这一步看到 `100.x.x.x` 很正常，但不要把它填进 ChatGPT。`100.x` 是 Tailnet 内网地址，不是公网 MCP 地址。

接着在 Tailscale 管理后台确认这台设备的完整 DNS 名称，例如：

```text
devspace-wsl.your-tailnet.ts.net
```

从下一步开始，用这个真实域名替换全文占位符。**到这里仍然没有打开 Funnel，只是先拿到 DevSpace 需要的公网 Origin。**

## 第三步：配置 DevSpace，并完成本地 OAuth 验收

编辑：

```text
~/.devspace/config.json
```

核心配置类似这样：

```json
{
  "host": "127.0.0.1",
  "port": 7676,
  "allowedRoots": [
    "/home/YOUR_WSL_USER/projects/example"
  ],
  "publicBaseUrl": "https://devspace-wsl.your-tailnet.ts.net",
  "allowedHosts": [
    "localhost",
    "127.0.0.1",
    "::1",
    "devspace-wsl.your-tailnet.ts.net"
  ]
}
```

`publicBaseUrl` 只写公网 Origin，不带 `/mcp`。ChatGPT 里填写连接地址时，才加 `/mcp`。

### 用 systemd 固定第一步验证过的启动命令

创建服务文件：

```bash
sudo nano /etc/systemd/system/devspace.service
```

写入：

```ini
[Unit]
Description=DevSpace MCP server
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=YOUR_WSL_USER
WorkingDirectory=/home/YOUR_WSL_USER
Environment=DEVSPACE_TOOL_MODE=codex
Environment=DEVSPACE_TRUST_PROXY=1
ExecStart=YOUR_DEVSPACE_START_COMMAND
Restart=always
RestartSec=2

[Install]
WantedBy=multi-user.target
```

`ExecStart` 要替换成第一步已经验证成功的绝对命令，例如：

```text
ExecStart=/absolute/path/to/devspace serve
```

或者：

```text
ExecStart=/absolute/path/to/node /absolute/path/to/start-script.mjs
```

如果 `devspace` 本身依赖一个不在 systemd 默认 PATH 里的 Node，不要继续赌环境变量。优先改成“绝对 Node 路径 + 绝对脚本路径”的启动方式。

启用服务：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now devspace.service
```

先确认服务真的在运行：

```bash
systemctl status devspace.service --no-pager
```

### 先看 `/mcp` 是否可达

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  http://127.0.0.1:7676/mcp
```

预期：

```text
401
```

这个 `401` 是好结果：说明 `/mcp` 存在，只是当前请求还没有授权。

### 再看 OAuth metadata 是否完整

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  http://127.0.0.1:7676/.well-known/oauth-protected-resource/mcp

curl -sS -o /dev/null -w '%{http_code}\n' \
  http://127.0.0.1:7676/.well-known/oauth-authorization-server/mcp
```

理想结果是：

```text
200
200
```

只有这三个本地信号全部成立，才继续打开公网：

```text
/mcp -> 401
protected resource metadata -> 200
authorization server metadata -> 200
```

### 如果 authorization server metadata 不是 200

不要先去排查 Funnel，也不要直接在 ChatGPT 里反复重连。先处理 DevSpace 本地 OAuth discovery。

优先顺序：

1. 先确认当前 DevSpace 版本是否已经原生提供这个路由。
2. 如果当前安装方式仍然缺少该路由，再加一个很薄的本地启动包装。

这个包装不负责公网代理，只是在 DevSpace app 开始监听前补上。下面只展示兼容路由的核心逻辑，不是完整启动脚本：

```js
const publicBaseUrl = config.publicBaseUrl.replace(/\/+$/, '');

app.get('/.well-known/oauth-authorization-server/mcp', (_req, res) => {
  res.setHeader('Access-Control-Allow-Origin', '*');
  res.json({
    issuer: `${publicBaseUrl}/`,
    authorization_endpoint: `${publicBaseUrl}/authorize`,
    response_types_supported: ['code'],
    code_challenge_methods_supported: ['S256'],
    token_endpoint: `${publicBaseUrl}/token`,
    grant_types_supported: ['authorization_code', 'refresh_token'],
    scopes_supported: config.oauth.scopes,
    revocation_endpoint: `${publicBaseUrl}/revoke`,
    registration_endpoint: `${publicBaseUrl}/register`,
  });
});
```

这段路由必须注册在 `app.listen(...)` 之前。至于 `config` 和 `app` 怎么导入，要按你的 DevSpace 安装方式处理，不要照抄别人的 Node 目录或包路径。

如果改成包装脚本启动，记得同步把 systemd 的 `ExecStart` 改成真实脚本命令，然后重新验收本地三个信号。

## 第四步：开启 Funnel 指向 DevSpace

直接把 Funnel 指向 DevSpace 的本地端口：

```bash
tailscale funnel --bg --yes 7676
```

查看状态：

```bash
tailscale funnel status
```

目标结构类似：

```text
https://devspace-wsl.your-tailnet.ts.net
|-- / proxy http://127.0.0.1:7676
```

如果命令提示需要在 Tailscale 管理后台启用 Funnel，就先到后台打开 Funnel 权限，再回到 WSL 重新执行上面的命令。

## 第五步：验证公网和 OAuth metadata

公网 `/mcp` 应该返回 `401`：

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  https://devspace-wsl.your-tailnet.ts.net/mcp
```

预期：

```text
401
```

再验证 OAuth protected resource：

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  https://devspace-wsl.your-tailnet.ts.net/.well-known/oauth-protected-resource/mcp
```

预期：

```text
200
```

再验证 authorization server metadata：

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  https://devspace-wsl.your-tailnet.ts.net/.well-known/oauth-authorization-server/mcp
```

预期：

```text
200
```

这一步只是把第三步的三个本地信号搬到公网再验收一次。如果本地已经是 `401 / 200 / 200`，公网却不是，就只查 Funnel、域名和 Host 校验，不要回头改 DevSpace 业务逻辑。

刚启用 Funnel 时，如果公网域名暂时还解析不到，先等几分钟再测；不要因为一次 DNS 失败就立刻重装 Tailscale。

## 第六步：在 ChatGPT 中连接

在 ChatGPT 的 MCP / Connector 配置里填写：

```text
https://devspace-wsl.your-tailnet.ts.net/mcp
```

认证方式选择：

```text
OAuth
```

不要填写：

```text
http://100.x.x.x:7676/mcp
http://127.0.0.1:7676/mcp
https://devspace-wsl.your-tailnet.ts.net
```

最后做一次真实调用验收：在 `allowedRoots` 里创建一个测试文件，让 ChatGPT 读取它。能读到文件，才说明网络、OAuth、DevSpace 权限和项目根目录全部闭环。
