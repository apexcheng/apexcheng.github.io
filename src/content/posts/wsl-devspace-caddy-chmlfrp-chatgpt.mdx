这篇文章只做一件事：把 WSL 里的 DevSpace 通过 Caddy 和 ChmlFrp 暴露成公网 HTTPS 地址，然后接入 ChatGPT。

本文假设你已经安装好 DevSpace、Caddy 和 ChmlFrp，并且已经准备好 ChmlFrp 隧道、独立域名和对应的 DNS 解析。下面只讲配置和验证，可以从上往下直接照抄。

## 为什么我选择这套方案

Cloudflare Tunnel 确实更省事，但对我这种「个人开发机 + WSL + DevSpace」场景，核心需求不是 CDN 或全球加速，而是让 ChatGPT 直接使用本机项目，同时把代码、运行环境和网络链路掌握在自己手里。WSL 提供 Linux 环境，DevSpace 让 ChatGPT 获得读取、搜索、修改和执行项目的能力，Caddy + ChmlFrp 负责提供公网入口。这样 VS Code、DevSpace 和 ChatGPT 操作的是同一份本地代码，链路也更直接。**简单说：AI 获得了操作能力，但代码和运行环境仍然在自己的机器上。**

<HighlightBox eyebrow="目标" title="最终给 ChatGPT 的地址只有一个">
  `https://your-domain.example.com/mcp`
</HighlightBox>

## 开始前先替换这些占位符

全文统一使用以下占位符。复制配置后，把它们替换成自己的真实值。

| 占位符 | 替换内容 |
| --- | --- |
| `YOUR_WSL_USER` | WSL 用户名 |
| `YOUR_CHMLFRP_USER` | ChmlFrp 用户 ID |
| `YOUR_CHMLFRP_TOKEN` | ChmlFrp Token |
| `YOUR_CHMLFRP_NODE_IP` | ChmlFrp 节点 IP |
| `YOUR_TUNNEL_NAME` | ChmlFrp 面板中的隧道名称 |
| `your-domain.example.com` | 自己的公网域名 |
| `/home/YOUR_WSL_USER/projects/example` | 想让 DevSpace 访问的项目目录 |

<Callout title="安全提醒">
  不要把真实 `token`、用户 ID、私有域名或不需要暴露的本地路径提交到公开仓库。
</Callout>

## 第一步：配置 WSL

编辑：

```text
/etc/wsl.conf
```

写入：

```ini
[boot]
systemd=true

[interop]
appendWindowsPath = true
```

保存后，在 Windows PowerShell 中完全关闭 WSL：

```powershell
wsl --shutdown
```

然后重新打开 Ubuntu。

检查 systemd 是否可用：

```bash
systemctl is-system-running
```

`appendWindowsPath` 必须放在 `[interop]` 下，不要写进 `[boot]`。

## 第二步：准备 Hermes Node，并初始化 DevSpace

这台机器的 DevSpace 安装在 Hermes 自带的 Node 环境里：

```text
/home/YOUR_WSL_USER/.hermes/node/bin/node
```

先把它临时加入当前 Shell 的 `PATH`：

```bash
export PATH="$HOME/.hermes/node/bin:$PATH"
```

检查 DevSpace：

```bash
devspace --version
```

本文实际验证版本：

```text
1.0.4
```

然后初始化 DevSpace：

```bash
devspace init
```

完成初始化后确认这两个文件都存在：

```bash
ls ~/.devspace/config.json ~/.devspace/auth.json
```

必须看到：

```text
~/.devspace/config.json
~/.devspace/auth.json
```

<Callout title="为什么这里先做 init？">
  `config.json` 只是服务配置；当前 DevSpace 还会使用 `auth.json` 保存认证相关信息。只手工创建 `config.json`，并不能保证新环境已经完成初始化。
</Callout>

`auth.json` 属于私密文件，不要提交到公开仓库。

## 第三步：配置并启动 DevSpace

编辑：

```text
~/.devspace/config.json
```

完整配置：

```json
{
  "host": "127.0.0.1",
  "port": 7676,
  "allowedRoots": [
    "/home/YOUR_WSL_USER/projects/example"
  ],
  "publicBaseUrl": "https://your-domain.example.com",
  "allowedHosts": [
    "localhost",
    "127.0.0.1",
    "::1",
    "your-domain.example.com"
  ]
}
```

这里主要修改三处：

1. `YOUR_WSL_USER`
2. `allowedRoots` 里的项目目录
3. `your-domain.example.com`

<Callout title="publicBaseUrl 怎么写">
  正确写法是 `https://your-domain.example.com`。不要加 `/mcp`，也不要写成本地的 `http://127.0.0.1:7676`。
</Callout>

### 创建当前机器实际使用的启动文件

当前这台机器的 DevSpace `1.0.4` 使用下面这个文件常驻运行。

新建：

```text
~/.devspace/serve-direct.mjs
```

写入：

```js
import { loadConfig } from "/home/YOUR_WSL_USER/.hermes/node/lib/node_modules/@waishnav/devspace/dist/config.js";
import { createServer } from "/home/YOUR_WSL_USER/.hermes/node/lib/node_modules/@waishnav/devspace/dist/server.js";
import { formatLocalAgentProviderAvailabilitySummary } from "/home/YOUR_WSL_USER/.hermes/node/lib/node_modules/@waishnav/devspace/dist/local-agent-availability.js";

const config = loadConfig();
const { app, close, localAgentProviders } = createServer(config);
const keepAlive = setInterval(() => {}, 60_000);

const httpServer = app.listen(config.port, config.host, () => {
  console.log(`devspace listening on http://${config.host}:${config.port}/mcp`);
  console.log(`public base url: ${config.publicBaseUrl}`);
  console.log(`allowed roots: ${config.allowedRoots.join(", ")}`);
  console.log(`allowed hosts: ${config.allowedHosts.join(", ")}`);
  console.log("auth: Owner password approval required");
  console.log(`logging: ${config.logging.level} ${config.logging.format}`);
  if (config.subagents) {
    console.log(`subagent providers: ${formatLocalAgentProviderAvailabilitySummary(localAgentProviders)}`);
  }
});

const shutdown = () => {
  clearInterval(keepAlive);
  httpServer.close(() => {
    close();
    process.exit(0);
  });
};

process.once("SIGINT", shutdown);
process.once("SIGTERM", shutdown);
```

把其中所有 `YOUR_WSL_USER` 替换成真实用户名。

<Callout title="这个文件和版本有关">
  它引用了 DevSpace `1.0.4` 的内部模块路径，是当前机器已经实际运行成功的方案。以后升级 DevSpace 后，如果 `dist/` 目录结构变化，需要重新验证。
</Callout>

### 配置 DevSpace systemd 服务

新建：

```text
/etc/systemd/system/devspace.service
```

写入：

```ini
[Unit]
Description=DevSpace MCP server
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=YOUR_WSL_USER
WorkingDirectory=/home/YOUR_WSL_USER
Environment=DEVSPACE_TOOL_MODE=codex
Environment=DEVSPACE_WIDGETS=changes
Environment=DEVSPACE_LOG_FORMAT=pretty
Environment=DEVSPACE_TRUST_PROXY=1
ExecStart=/home/YOUR_WSL_USER/.hermes/node/bin/node /home/YOUR_WSL_USER/.devspace/serve-direct.mjs
Restart=always
RestartSec=2

[Install]
WantedBy=multi-user.target
```

这里故意使用 Node 的绝对路径。systemd 不依赖交互式 Shell 的 `PATH`，这样最稳定。

执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now devspace.service
```

检查状态：

```bash
systemctl status devspace.service --no-pager -l
```

目标：

```text
active (running)
```

### 验收 DevSpace：7676 必须返回 401

执行：

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  http://127.0.0.1:7676/mcp
```

预期：

```text
401
```

这个 `401` 是好结果：DevSpace 已经启动，`/mcp` 路径存在，只是当前请求没有认证。

## 第四步：配置并验收 Caddy

编辑：

```text
/etc/caddy/Caddyfile
```

写入：

```caddyfile
{
    https_port 7677
}

https://your-domain.example.com {
    bind 127.0.0.1
    reverse_proxy 127.0.0.1:7676
}
```

这段配置只做一件事：

```text
Caddy 127.0.0.1:7677 / HTTPS
                ↓
DevSpace 127.0.0.1:7676 / HTTP
```

重启并启用 Caddy：

```bash
sudo systemctl restart caddy
sudo systemctl enable caddy
```

检查状态：

```bash
systemctl status caddy.service --no-pager -l
```

目标状态：

```text
active (running)
```

### 确认 7676 和 7677 都在监听

执行：

```bash
ss -ltn | grep -E ':(7676|7677)\b'
```

应该同时看到：

```text
127.0.0.1:7676
127.0.0.1:7677
```

### 验收 Caddy：本地 HTTPS 必须返回 401

执行：

```bash
curl -sk \
  --resolve your-domain.example.com:7677:127.0.0.1 \
  -o /dev/null \
  -w '%{http_code}\n' \
  https://your-domain.example.com:7677/mcp
```

预期：

```text
401
```

做到这里，可以确认：

```text
DevSpace 7676 正常
+
Caddy 7677 HTTPS 正常
```

如果这一步还没成功，不要先排查 ChmlFrp。

## 第五步：配置并验收 ChmlFrp

编辑：

```text
~/.chmlfrp/frpc.ini
```

完整模板：

```ini
[common]
server_addr = YOUR_CHMLFRP_NODE_IP
server_port = 7000
tls_enable = false
login_fail_exit = false
user = YOUR_CHMLFRP_USER
token = YOUR_CHMLFRP_TOKEN

[YOUR_TUNNEL_NAME]
type = https
local_ip = 127.0.0.1
local_port = 7677
custom_domains = your-domain.example.com
```

最关键的是这两行：

```ini
type = https
local_port = 7677
```

这里的 `local_port` 必须指向 Caddy，不要指向 DevSpace。

另外：

```ini
[YOUR_TUNNEL_NAME]
```

必须替换成 ChmlFrp 面板里的真实隧道名称。

新建：

```text
/etc/systemd/system/chmlfrp-frpc.service
```

写入：

```ini
[Unit]
Description=ChmlFrp FRPC
After=network.target

[Service]
Type=simple
User=YOUR_WSL_USER
WorkingDirectory=/home/YOUR_WSL_USER/.chmlfrp
ExecStart=/home/YOUR_WSL_USER/.chmlfrp/frpc -c /home/YOUR_WSL_USER/.chmlfrp/frpc.ini
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

把 `YOUR_WSL_USER` 替换后执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now chmlfrp-frpc.service
```

检查：

```bash
systemctl status chmlfrp-frpc.service --no-pager -l
```

再看最近日志：

```bash
journalctl -u chmlfrp-frpc.service -n 50 --no-pager -l
```

正常时应该看到类似：

```text
login to server success
proxy added: [YOUR_TUNNEL_NAME]
[YOUR_TUNNEL_NAME] start proxy success
```

### 验收公网 MCP：必须返回 401

执行：

```bash
curl -sS -o /dev/null -w '%{http_code}\n' \
  https://your-domain.example.com/mcp
```

预期：

```text
401
```

如果公网也返回 `401`，说明这条链路已经真正打通：

```text
公网 HTTPS
→ ChmlFrp
→ Caddy 7677
→ DevSpace 7676
```

### 再验证 OAuth metadata

执行：

```bash
curl -sS https://your-domain.example.com/.well-known/oauth-protected-resource/mcp
```

正常时应该返回类似：

```json
{
  "resource": "https://your-domain.example.com/mcp",
  "authorization_servers": ["https://your-domain.example.com/"],
  "scopes_supported": ["devspace"],
  "resource_name": "DevSpace"
}
```

## 第六步：让 Windows 登录后保持 WSL 运行

WSL 退出后，里面的 DevSpace、Caddy 和 frpc 都会停止。

最简单的做法是在 Windows 启动目录放一个 VBS 脚本，让 WSL 在登录后保持运行。

文件路径：

```text
%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\DevSpace-WSL-KeepAlive.vbs
```

文件内容：

```vbscript
Set shell = CreateObject("WScript.Shell")
shell.Run """C:\Windows\System32\wsl.exe"" -d Ubuntu --exec sleep infinity", 0, False
```

这里的：

```text
Ubuntu
```

要和下面命令显示的发行版名称一致：

```powershell
wsl -l -v
```

例如看到：

```text
NAME      STATE      VERSION
Ubuntu    Running    2
```

那 VBS 中就继续使用：

```text
-d Ubuntu
```

## 第七步：一次性检查所有服务

先检查 WSL：

```powershell
wsl -l -v
```

目标：

```text
Ubuntu    Running    2
```

再检查三个 systemd 服务：

```bash
systemctl status devspace.service caddy.service chmlfrp-frpc.service --no-pager -l
```

三个都应该是：

```text
active (running)
```

## 第八步：在 ChatGPT 中连接，并完成实际调用验收

最终填写：

```text
https://your-domain.example.com/mcp
```

不要填写：

```text
https://your-domain.example.com:7677/mcp
```

也不要填写：

```text
http://127.0.0.1:7676/mcp
```

ChatGPT 只需要公网 HTTPS 地址。

填写地址后，完成 DevSpace 的授权流程。

然后在 `allowedRoots` 中准备一个最简单的测试文件：

```bash
printf 'devspace-ok\n' > \
  /home/YOUR_WSL_USER/projects/example/DEVSPACE_VERIFY.txt
```

接着让 ChatGPT 通过 DevSpace 读取：

```text
DEVSPACE_VERIFY.txt
```

预期内容：

```text
devspace-ok
```

<HighlightBox eyebrow="SUCCESS" title="读到测试文件，才算真正配置成功">
  公网 `/mcp` 返回 `401` 只能证明网络链路和认证入口可达；ChatGPT 能实际读取 `allowedRoots` 中的文件，才证明整条接入链路真正可用。
</HighlightBox>

## 常见的坑

正文配置照抄后，如果仍然连不上，优先检查下面几项。

### 先检查这三个本机实际遇到的问题

#### `devspace` 提示找不到 Node

当前这台机器直接执行：

```bash
devspace --version
```

可能报错：

```text
/usr/bin/env: ‘node’: No such file or directory
```

先执行：

```bash
export PATH="$HOME/.hermes/node/bin:$PATH"
```

再运行 `devspace --version`。

#### `serve-direct.mjs` 不存在

如果 systemd 日志提示找不到：

```text
~/.devspace/serve-direct.mjs
```

说明漏掉了本文第三步里的启动文件。

检查：

```bash
ls ~/.devspace/serve-direct.mjs
```

#### `auth.json` 不存在

检查：

```bash
ls ~/.devspace/auth.json
```

如果不存在，先确认已经执行：

```bash
devspace init
```

不要只手工创建 `config.json` 就直接启动服务。
