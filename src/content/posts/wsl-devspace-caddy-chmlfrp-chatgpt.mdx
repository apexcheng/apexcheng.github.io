这篇文章只做一件事：把 WSL 里的 DevSpace 通过 Caddy 和 ChmlFrp 暴露成公网 HTTPS 地址，然后接入 ChatGPT。

本文假设你已经安装好 DevSpace、Caddy 和 ChmlFrp，并且已经准备好 ChmlFrp 隧道、独立域名和对应的 DNS 解析。下面只讲配置和验证，可以从上往下直接照抄。

<HighlightBox eyebrow="目标" title="最终给 ChatGPT 的地址只有一个">
  `https://your-domain.example.com/mcp`
</HighlightBox>

## 最终架构

先看清楚整条链路，后面的所有配置都围绕这四层展开。

<Mermaid
  title="ChatGPT 到 WSL 的最终连接链路"
  caption="公网 URL 不带本地端口；ChmlFrp 连接 Caddy，Caddy 再转发到 DevSpace。"
  chart={`flowchart LR
    A[ChatGPT] -->|HTTPS /mcp| B[ChmlFrp 公网入口]
    B -->|HTTPS 隧道| C[Caddy 127.0.0.1:7677]
    C -->|HTTP 反向代理| D[DevSpace 127.0.0.1:7676]`}
/>

这几个地址不要混：

| 位置 | 地址 | 协议 |
| --- | --- | --- |
| ChatGPT 使用 | `https://your-domain.example.com/mcp` | HTTPS |
| Caddy 本地监听 | `127.0.0.1:7677` | HTTPS |
| DevSpace 本地监听 | `127.0.0.1:7676` | HTTP |

<Callout title="最容易记错的一点">
  `7677` 只是 WSL 里的本地端口。ChatGPT 的公网地址不要写成 `https://your-domain.example.com:7677/mcp`。
</Callout>

## 开始前先替换这些占位符

全文统一使用以下占位符。复制配置后，把它们替换成自己的真实值。

| 占位符 | 替换内容 |
| --- | --- |
| `YOUR_WSL_USER` | WSL 用户名 |
| `YOUR_CHMLFRP_USER` | ChmlFrp 用户 ID |
| `YOUR_CHMLFRP_TOKEN` | ChmlFrp Token |
| `YOUR_CHMLFRP_NODE_IP` | ChmlFrp 节点 IP |
| `YOUR_TUNNEL_NAME` | ChmlFrp 面板中的隧道名称 |
| `your-domain.example.com` | 自己的公网域名 |
| `/home/YOUR_WSL_USER/projects/example` | 想让 DevSpace 访问的项目目录 |

<Callout title="安全提醒">
  不要把真实 `token`、用户 ID、私有域名或不需要暴露的本地路径提交到公开仓库。
</Callout>

## 第一步：配置 WSL

编辑：

```text
/etc/wsl.conf
```

写入：

```ini
[boot]
systemd=true

[interop]
appendWindowsPath = true
```

保存后，在 Windows PowerShell 中完全关闭 WSL：

```powershell
wsl --shutdown
```

然后重新打开 Ubuntu。

检查 systemd 是否可用：

```bash
systemctl is-system-running
```

`appendWindowsPath` 必须放在 `[interop]` 下，不要写进 `[boot]`。

## 第二步：配置 DevSpace

编辑：

```text
~/.devspace/config.json
```

完整配置：

```json
{
  "host": "127.0.0.1",
  "port": 7676,
  "allowedRoots": [
    "/home/YOUR_WSL_USER/projects/example"
  ],
  "publicBaseUrl": "https://your-domain.example.com",
  "allowedHosts": [
    "localhost",
    "127.0.0.1",
    "::1",
    "your-domain.example.com"
  ]
}
```

这里主要修改三处：

1. `YOUR_WSL_USER`
2. `allowedRoots` 里的项目目录
3. `your-domain.example.com`

<Callout title="publicBaseUrl 怎么写">
  正确写法是 `https://your-domain.example.com`。不要加 `/mcp`，也不要写成本地的 `http://127.0.0.1:7676`。
</Callout>

## 第三步：配置 Caddy

编辑：

```text
/etc/caddy/Caddyfile
```

写入：

```caddyfile
{
    https_port 7677
}

https://your-domain.example.com {
    bind 127.0.0.1
    reverse_proxy 127.0.0.1:7676
}
```

这段配置只做一件事：

```text
Caddy 127.0.0.1:7677 / HTTPS
                ↓
DevSpace 127.0.0.1:7676 / HTTP
```

重启并启用 Caddy：

```bash
sudo systemctl restart caddy
sudo systemctl enable caddy
```

检查状态：

```bash
systemctl status caddy.service --no-pager -l
```

目标状态：

```text
active (running)
```

## 第四步：配置 ChmlFrp

编辑：

```text
~/.chmlfrp/frpc.ini
```

完整模板：

```ini
[common]
server_addr = YOUR_CHMLFRP_NODE_IP
server_port = 7000
tls_enable = false
login_fail_exit = false
user = YOUR_CHMLFRP_USER
token = YOUR_CHMLFRP_TOKEN

[YOUR_TUNNEL_NAME]
type = https
local_ip = 127.0.0.1
local_port = 7677
custom_domains = your-domain.example.com
```

最关键的是这两行：

```ini
type = https
local_port = 7677
```

这里的 `local_port` 必须指向 Caddy，不要指向 DevSpace。

另外：

```ini
[YOUR_TUNNEL_NAME]
```

必须替换成 ChmlFrp 面板里的真实隧道名称。

## 第五步：把 DevSpace 交给 systemd

新建：

```text
/etc/systemd/system/devspace.service
```

写入：

```ini
[Unit]
Description=DevSpace MCP server
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=YOUR_WSL_USER
WorkingDirectory=/home/YOUR_WSL_USER
Environment=DEVSPACE_TOOL_MODE=codex
Environment=DEVSPACE_WIDGETS=changes
Environment=DEVSPACE_LOG_FORMAT=pretty
Environment=DEVSPACE_TRUST_PROXY=1
ExecStart=/home/YOUR_WSL_USER/.hermes/node/bin/node /home/YOUR_WSL_USER/.devspace/serve-direct.mjs
Restart=always
RestartSec=2

[Install]
WantedBy=multi-user.target
```

把 `YOUR_WSL_USER` 替换成真实用户名，然后执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now devspace.service
```

检查：

```bash
systemctl status devspace.service --no-pager -l
```

目标状态：

```text
active (running)
```

## 第六步：把 ChmlFrp 交给 systemd

新建：

```text
/etc/systemd/system/chmlfrp-frpc.service
```

写入：

```ini
[Unit]
Description=ChmlFrp FRPC
After=network-online.target devspace.service caddy.service
Wants=network-online.target devspace.service caddy.service

[Service]
Type=simple
User=YOUR_WSL_USER
WorkingDirectory=/home/YOUR_WSL_USER
ExecStart=/home/YOUR_WSL_USER/.chmlfrp/frpc -c /home/YOUR_WSL_USER/.chmlfrp/frpc.ini
Restart=always
RestartSec=1

[Install]
WantedBy=multi-user.target
```

把 `YOUR_WSL_USER` 替换后执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now chmlfrp-frpc.service
```

检查：

```bash
systemctl status chmlfrp-frpc.service --no-pager -l
```

再看最近日志：

```bash
journalctl -u chmlfrp-frpc.service -n 50 --no-pager -l
```

正常时应该看到类似：

```text
login to server success
proxy added: [YOUR_TUNNEL_NAME]
[YOUR_TUNNEL_NAME] start proxy success
```

## 第七步：让 Windows 登录后保持 WSL 运行

WSL 退出后，里面的 DevSpace、Caddy 和 frpc 都会停止。

最简单的做法是在 Windows 启动目录放一个 VBS 脚本，让 WSL 在登录后保持运行。

文件路径：

```text
%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\DevSpace-WSL-KeepAlive.vbs
```

文件内容：

```vbscript
Set shell = CreateObject("WScript.Shell")
shell.Run """C:\Windows\System32\wsl.exe"" -d Ubuntu --exec sleep infinity", 0, False
```

这里的：

```text
Ubuntu
```

要和下面命令显示的发行版名称一致：

```powershell
wsl -l -v
```

例如看到：

```text
NAME      STATE      VERSION
Ubuntu    Running    2
```

那 VBS 中就继续使用：

```text
-d Ubuntu
```

## 第八步：一次性检查所有服务

先检查 WSL：

```powershell
wsl -l -v
```

目标：

```text
Ubuntu    Running    2
```

再检查三个 systemd 服务：

```bash
systemctl status devspace.service caddy.service chmlfrp-frpc.service --no-pager -l
```

三个都应该是：

```text
active (running)
```

## 第九步：验证公网 OAuth 地址

执行：

```bash
curl -sS https://your-domain.example.com/.well-known/oauth-protected-resource/mcp
```

正常时应该返回类似：

```json
{
  "resource": "https://your-domain.example.com/mcp",
  "authorization_servers": ["https://your-domain.example.com/"],
  "scopes_supported": ["devspace"],
  "resource_name": "DevSpace"
}
```

只要这里能正常返回 JSON，就说明公网 HTTPS 已经能到达 DevSpace。

## 第十步：验证 MCP 地址

执行：

```bash
curl -sS https://your-domain.example.com/mcp
```

未带认证信息时，预期返回类似：

```json
{
  "error": "invalid_token",
  "error_description": "Missing Authorization header"
}
```

<HighlightBox eyebrow="验证结果" title="这个 401 是正常的">
  它说明 DNS、HTTPS、ChmlFrp、Caddy 和 DevSpace 已经连通。当前请求只是没有携带 Authorization，所以被 DevSpace 拒绝。
</HighlightBox>

## 第十一步：在 ChatGPT 中填写地址

最终填写：

```text
https://your-domain.example.com/mcp
```

不要填写：

```text
https://your-domain.example.com:7677/mcp
```

也不要填写：

```text
http://127.0.0.1:7676/mcp
```

ChatGPT 只需要公网 HTTPS 地址。

## 最终检查清单

<HighlightBox eyebrow="CHECK" title="全部满足后，再去 ChatGPT 连接">
  <ul>
    <li>DevSpace 使用 `127.0.0.1:7676`，协议是 HTTP。</li>
    <li>Caddy 使用 `127.0.0.1:7677`，协议是 HTTPS。</li>
    <li>ChmlFrp 使用 `type = https`。</li>
    <li>ChmlFrp 使用 `local_port = 7677`。</li>
    <li>`publicBaseUrl` 是公网 HTTPS Origin，不带 `/mcp`。</li>
    <li>ChatGPT 地址是 `https://your-domain.example.com/mcp`。</li>
    <li>公网地址不带 `:7677`。</li>
    <li>ChmlFrp 隧道名称和 `frpc.ini` section 名称一致。</li>
    <li>DevSpace、Caddy、frpc 三个服务都是 `active (running)`。</li>
    <li>WSL 状态是 `Running`。</li>
    <li>OAuth metadata 能返回 JSON。</li>
    <li>直接请求 `/mcp` 能返回 `invalid_token` 或缺少 Authorization 的 401。</li>
  </ul>
</HighlightBox>

这套配置真正需要记住的只有一条：公网 HTTPS 先到 ChmlFrp，再进入 Caddy 的 7677，最后由 Caddy 转发到 DevSpace 的 7676。端口和协议不要混，整条链路就清楚了。
