## 我为什么整理这个项目

影刀可视化指令很直观，但进入编码版以后，很多问题不能只靠普通 Python 经验解决。

| 实际遇到的问题 | 这个项目提供什么 |
| --- | --- |
| 不知道影刀 API 应该怎么调用 | `xbot-api-docs/docs/` 中的专项文档和最小示例 |
| Agent 容易把 Selenium、Playwright、openpyxl 的写法套进影刀 | `AGENTS.md` 中的开发边界和禁止事项 |
| 不知道当前问题应该查哪篇文档 | `llms.txt` 中的文档入口和搜索意图 |
| 市场指令在可视化界面能运行，编码版却不知道参数 | 市场指令参数映射与源码排查记录 |
| 外部编辑器已经改完代码，影刀里却没有对应流程 | `shadowbot_sync_tool.py` 同步工具 |
| 同一个坑过一段时间又重新踩一次 | `wiki/error-book.md` 和 `wiki/unresolved.md` |

这个项目不是一个可以直接运行的影刀应用，也不是把所有影刀源码搬进仓库。它更像我自己的影刀开发手册：把已经确认的 API、实际踩过的坑、Agent 规则和同步方式放在一起，后面开发时可以直接查。

项目地址：**[apexcheng/yingdao-xbot-ai-agent](https://github.com/apexcheng/yingdao-xbot-ai-agent)**

只把知识库交给 Agent，它最多能回答问题；只把真实项目交给 Agent，它又可能因为缺少影刀上下文而猜 API。

正确关系很简单：**知识库负责提供依据，真实项目负责承载代码。**

## 先让 Agent 帮我回答影刀问题

不需要一开始就让 Agent 修改项目。最轻量的用法，是先让它查一个明确问题。

例如我要确认：影刀编码版怎样把二维数组批量写入 Excel。

```text
影刀编码版怎么把二维数组批量写入 Excel？

请按下面顺序处理：
1. 先检查当前项目已有代码。
2. 当前项目无法确认时，再查 yingdao-xbot-ai-agent。
3. 不要使用普通 openpyxl 写法替代 xbot API。
4. 给出最小示例和依据文件路径。
5. 没有在真实影刀环境验证的部分，标记“需运行验证”。
```

这个提问里真正有用的是三件事：

1. **先看当前项目。** 已有代码往往最接近现在的影刀版本和业务现场。
2. **给出依据路径。** 可以快速判断答案是查到的，还是模型凭经验推测的。
3. **区分已确认与待验证。** 文档、源码和真实运行不是同一个证据等级。

同样的写法也可以用于报错排查：

```text
这是完整报错：
[粘贴错误信息]

先检查当前项目，再查知识库中的排错记录。
一次只验证一个假设，不要同时修改多个参数。
保留完整错误，不要静默吞异常。
```

## 再让 Agent 修改真实影刀项目

现在回到本文的真实任务：

> 从 Excel 读取订单号，打开网页逐行查询订单状态，再把结果写回原表。

先不要说“帮我写一个影刀流程”，而是把输入、操作、输出和异常说清楚。

```text
业务目标：
读取 Excel 中的订单号，逐行打开网页查询订单状态，
并把查询结果写回原表指定列。

验收标准：
- 跳过空订单号
- 每个订单只查询一次
- 查询结果写入指定列
- 页面异常时记录错误，不覆盖原数据
- 最后回到影刀编辑器实际运行一次
```

接下来，Agent 才能沿着一条清楚的路径工作。

### 第一步：先读现有代码

Agent 应先确认：

- 项目入口是 `main(args)`、`run.py`，还是其他函数；
- Excel 对象和浏览器对象从哪里传入；
- 当前项目已经使用哪些 xbot API；
- 日志是否使用 `from xbot.app import logging`；
- 订单号列、结果列和异常列现在叫什么。

能从当前代码确认的事情，不需要为了形式再查一遍知识库。

### 第二步：只查当前不确定的 API

例如当前项目已经有读取 Excel 的代码，但没有网页查询，那么只查浏览器对象、元素输入、点击、等待和结果读取，不要把整套知识库全部加载进上下文。

查询顺序可以保持为：

```text
当前项目已有代码
↓
xbot-api-docs/docs/ 对应文档
↓
相关示例或市场指令说明
↓
必要时查看允许读取的真实实现
↓
仍不确定：标记“需运行验证”
```

### 第三步：只改当前任务需要的地方

如果现有流程已经能打开 Excel、启动浏览器和保存文件，就不要顺手重写整个项目。当前任务只需要增加：

- 读取一行订单号；
- 在网页输入订单号并查询；
- 读取订单状态；
- 把状态写回对应行；
- 页面异常时记录错误并继续下一行。

这类修改更容易验证，也更容易发现到底是哪一步出了问题。

## 网页元素怎么找

网页自动化最难的通常不是 Python，而是当前页面到底是什么结构。

以订单状态查询为例，Agent 需要确认：

- 订单号输入框是否在 iframe 中；
- 查询按钮有没有稳定属性；
- 点击后应该等待页面跳转，还是等待局部区域刷新；
- 订单状态显示在普通文本、表格单元格还是弹窗里；
- 页面滚动和表格内部滚动是不是同一回事。

元素定位优先级可以保持为：

```text
稳定属性
↓
明确文本
↓
局部父子关系
↓
动态 class
↓
坐标点击
```

坐标点击不是不能用，但应该放在最后。页面尺寸、缩放比例或弹窗位置一变，坐标通常最先失效。

## 外部代码怎么同步回影刀

Agent 在真实项目目录改完 `.py` 文件后，还要让影刀重新识别这些文件和入口。

同步工具在知识库目录中，真实项目路径通过 `--project-dir` 指定：

```powershell
python "C:\path\to\yingdao-xbot-ai-agent\shadowbot_sync_tool.py" ^
  --project-dir "%LOCALAPPDATA%\ShadowBot\users\<user_id>\apps\<app_id>" ^
  prepare main.py
```

这三个状态必须分开说：

| 状态 | 代表什么 |
| --- | --- |
| 文件修改完成 | Agent 已经写完代码 |
| `prepare` 成功 | 文件登记和编译准备完成 |
| 影刀实际运行成功 | 页面操作、参数和业务结果全部通过 |

`prepare` 成功，不代表订单号一定输入到了正确页面，也不代表查询结果一定写回了正确行。最终验收仍然要回到影刀编辑器实际运行。

## 知识库怎么继续更新

知识库不是整理一次就结束。影刀版本、市场指令参数、网页结构和第三方接口都会变化。真实项目每解决一个新问题，都可能产生下一次可以直接复用的结论。

### 方式一：让 Agent 分析指令后补库

遇到官方内置指令或市场指令没有清楚的编码版说明时，可以让 Agent 分析允许查看的本地实现和调用信息。

例如：某个市场指令在可视化界面可以运行，但编码版不知道真实入口和参数名。

```text
这个市场指令在可视化界面可以运行，
但知识库里没有清楚的编码版说明。

请按下面步骤处理：
1. 先查现有文档和示例。
2. 必要时分析允许查看的本地实现。
3. 整理真实导入路径和调用入口。
4. 列出参数名、默认值、建议类型和返回值。
5. 编写一个最小示例。
6. 明确哪些来自源码，哪些已经运行验证，哪些仍需验证。
7. 把结论补充回知识库并更新索引。
```

Agent 最终应该整理出：

- 指令名称和适用场景；
- 真实导入路径与调用入口；
- 参数名、默认值和必要类型；
- 返回值和已知异常；
- 一个最小可用示例；
- 已验证、源码确认和待验证边界。

资料可以按内容写入：

```text
xbot-api-docs/docs/        API 和市场指令说明
xbot-api-docs/examples/    最小示例
llms.txt                   新文档查询入口
wiki/error-book.md         已确认的错误修正
wiki/unresolved.md         尚未完成运行验证的结论
```

最重要的边界是：**看过源码不等于已经在影刀中运行通过。** 来源不同，文档里的证据等级也要分开写。

### 方式二：直接把已知结论告诉 AI

如果我已经通过真实影刀运行确认某个参数或坑点，就不用让 Agent 再从头调查。

例如我已经确认：某个 `timeout` 参数单位是秒，不是毫秒；传字符串会报错，传整数可以正常运行。

可以直接告诉 Agent：

```text
我已经在真实影刀环境确认：
- timeout 的单位是秒，不是毫秒；
- 传入字符串会报类型错误；
- 传入整数可以正常运行。

请把这个结论补充进影刀知识库：
1. 写明正确用法；
2. 记录错误表现；
3. 增加最小示例；
4. 更新相关文档索引；
5. 不要扩展我没有确认的结论；
6. 仍不确定的部分放进待验证事项。
```

这种方式的重点不是让 AI 再证明一次，而是把已经确认的事实整理清楚：

- 已知结论直接沉淀；
- 不扩展没有确认的内容；
- 旧文档写错时同步更新错误修正记录；
- 仍不确定的内容继续留在待验证事项。

## 更新以后，下一次开发会发生什么

第一次遇到一个陌生问题时，通常要经历：

```text
查资料 → 尝试 → 报错 → 排查 → 运行验证 → 记录结论
```

下一次遇到相同问题，Agent 可以直接查到已经验证过的文档：

```text
读取当前项目 → 命中正确文档 → 使用已验证参数 → 完成最小修改
```

这才是知识库持续更新最实际的价值：不是让仓库看起来越来越大，而是让下一次开发少走一段已经走过的路。

## 一段可以直接使用的完整提示词
