## 这个项目到底解决什么问题

影刀可视化指令已经比较容易上手，但进入编码版开发后，问题会明显变多：API 资料分散、参数不够清楚、市场指令的界面参数和代码参数可能不同，代码在外部编辑器改完以后还要重新同步，最后还必须回到影刀环境确认真实业务结果。

如果没有专门的知识库，Agent 很容易把“看起来相似”的普通 Python 经验直接套进影刀：函数名似乎合理，参数也像那么回事，但一运行就会出现对象不存在、参数层级错误、返回值结构不符，或者代码改了却根本没有进入真实应用。

它不是影刀官方仓库，也不是一个可以直接运行的影刀应用。它更像一层“开发者翻译器”：把影刀 API、开发规则、历史错误、待验证事项和同步动作整理成 Agent 能理解、能搜索、能执行的工作流。

## 最重要的概念：必须同时告诉 Agent 两个目录

影刀 Agent 开发里最容易混淆的，不是语法，而是**知识库目录**和**真实影刀项目目录**。

只告诉 Agent 知识库目录，它最多能回答问题；只告诉它真实项目目录，它又可能因为缺少影刀上下文而猜 API。正确方式是：**左边提供规则和依据，右边提供真实代码和业务现场。**

## Agent 开工前，需要的不是一句“帮我写流程”

一个稳定的任务描述，至少要同时包含业务目标和开发上下文。

例如，不要只说：

> 帮我做一个查询订单状态的影刀流程。

更清楚的写法是：

```text
知识库目录：C:\path\to\yingdao-xbot-ai-agent
真实项目目录：C:\path\to\shadowbot-project

目标：读取指定 Excel 中的订单号，逐行打开网页查询状态，并写回原表。

验收标准：
- 跳过空订单号
- 每个订单只查询一次
- 结果写入指定列
- 页面异常时记录错误，不覆盖原数据
- 最后在影刀编辑器中完成一次实际运行验证

执行要求：
1. 先读取知识库中的 AGENTS.md 和 llms.txt。
2. 再读取真实项目现有代码，定位当前实现。
3. xbot API 必须以文档、现有代码或可验证源码为依据。
4. 只修改当前需求直接相关的内容。
5. 修改后执行同步工具，并说明影刀内验证结果。
```

这段提示词真正重要的不是格式，而是把“目标、目录、依据、修改范围、验收方式”一次讲清楚。

复杂任务可以创建 `TASK.md`，把调查、方案、实现、验证四个阶段和当前状态留下来；只改一个函数、参数或错误描述时，直接做最小修改，不需要为了“看起来正式”额外制造流程。

## 不同任务，应该查不同入口

`llms.txt` 的作用不是把所有文档一次塞给 Agent，而是让它根据任务按需进入正确入口。

这能减少一个常见问题：上下文越多，Agent 不一定越准确。一次加载全部文档，反而可能混用不相关 API、忽略当前项目规则，甚至重复生成项目里已经存在的能力。

## 浏览器探索，是网页自动化里的现场调查

网页自动化最难的通常不是 Python 语法，而是页面当前到底长什么样：按钮是否位于 iframe、元素是否被遮挡、滚动的是页面还是局部容器、点击后是否需要等待异步加载、登录状态或风控是否改变了页面。

当 Agent 可以通过授权的浏览器调试工具查看页面时，它就不必只根据截图猜测元素位置。

## 外部代码写完后，为什么还要同步

Agent 在真实项目目录修改 `.py` 文件以后，影刀不一定自动认识新增流程和辅助模块。项目根目录中的 `shadowbot_sync_tool.py` 用于把外部修改重新登记并准备编译。

```powershell
python "C:\path\to\yingdao-xbot-ai-agent\shadowbot_sync_tool.py" ^
  --project-dir "%LOCALAPPDATA%\ShadowBot\users\<user_id>\apps\<app_id>" ^
  prepare main.py
```

## 出错时，先判断问题属于哪一层

知识库的另一个价值，是不只记录“正确用法”，也记录怎样排错、过去哪些说法错了、还有哪些结论没有完成运行验证。

<div class="yxa-debug-board" aria-label="影刀问题分层排查">
  <div class="yxa-debug-board__head">
    <span>TROUBLESHOOTING STACK</span>
    <strong>先分层，再一次验证一个假设</strong>
    <p>不要同时改多个参数。否则即使问题消失，也不知道真正原因是什么。</p>
  </div>
  <div class="yxa-debug-board__layers">
    <div><b>L1</b><strong>业务输入层</strong><p>空值、字段名、Sheet、长数字、日期格式、重复数据和文件路径。</p><span>先看数据</span></div>
    <div><b>L2</b><strong>xbot API 层</strong><p>对象来源、参数名称、类型、层级、返回值和相似库误用。</p><span>查文档</span></div>
    <div><b>L3</b><strong>页面与环境层</strong><p>登录、风控、iframe、元素遮挡、WPS 进程和 Windows 窗口状态。</p><span>看现场</span></div>
    <div><b>L4</b><strong>市场指令与接口层</strong><p>参数映射、版本变化、请求体、服务端 500、限流和 unknownError。</p><span>留 request id</span></div>
    <div><b>L5</b><strong>Agent 上下文层</strong><p>错误目录、过期资料、上下文过多、把推测写成已验证、扩大修改范围。</p><span>回到规则</span></div>
  </div>
</div>

一个更可靠的排错顺序是：保留完整错误 → 构造最小复现 → 确认真实项目和运行环境 → 定位文件、函数与输入 → 查对应文档和示例 → 一次只验证一个假设 → 做最小修复 → 回到影刀重新运行。

## 知识库为什么需要持续更新

影刀版本、市场指令、网页结构和第三方接口都会变化。一次整理完成，并不代表以后永远正确。

<div class="yxa-loop" aria-label="影刀知识库更新闭环">
  <div><span>真实项目</span><strong>遇到新问题</strong><small>保留完整错误与现场</small></div>
  <i>→</i>
  <div><span>Agent / 人工</span><strong>查文档与最小复现</strong><small>区分推测、源码和运行结果</small></div>
  <i>→</i>
  <div class="is-core"><span>知识库</span><strong>修正文档与边界</strong><small>error-book / unresolved</small></div>
  <i>→</i>
  <div><span>下一次开发</span><strong>直接复用结论</strong><small>减少重复踩坑</small></div>
</div>

项目把结果分成三种状态：

- **已经在影刀环境运行通过**：可以记录为已验证；
- **只通过文档或源码确认**：必须写明依据；
- **仅根据现象推测**：放进待验证事项，不能写成确定结论。

错误结论也不应该静默覆盖。记录“旧描述是什么、为什么错、正确说法是什么、依据是什么”，可以避免下一个 Agent 或下一个项目继续复制同一个错误。
