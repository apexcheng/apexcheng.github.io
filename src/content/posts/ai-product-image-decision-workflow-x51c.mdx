这篇文章不准备讲 Prompt，也不准备写成 Skill 技术说明书。

我更想把现在这套 **AI 商品图项目到底怎么用、背后怎么运行、以后出问题去哪里查** 记录清楚。

实际案例还是 X51C 触控蓝牙键盘。

如果只站在运营视角，使用过程很简单：准备商品资料，告诉 AI 要做一套商品图，看它给出的整套方案，确认后再逐张生成。

但项目内部并不是“一句话直接出图”。它会检查资料、调用不同 Skill、生成状态文件、等待人工确认；如果缺少必须的产品角度，还会停下来让用户补拍照片。

## 作为运营，我实际上怎么使用

第一次使用，不需要先理解 Skill，也不需要知道项目目录。

只需要准备几类资料。

### 必须准备

```text
商品名称和型号
真实产品图
商品参数和功能
SKU / 套装信息
已知卖点
不能夸大的内容
目标平台
竞品链接
```

### 有的话更好

```text
用户评价
差评和追评
客服高频问题
旧版主图和详情图
目标人群和价格定位
```

然后直接告诉 AI：

```text
我要给 X51C 做一套主图和详情图。
```

从用户视角看，后面主要有四个节点。

```text
准备资料
↓
发起商品图任务
↓
确认整套图片计划
↓
逐张生成和审核
```

真正复杂的部分，项目自己在背后处理。

## 项目只有一个使用入口

整个项目里，我不希望运营自己判断：

```text
现在应该用竞品分析 Skill？
还是评价分析 Skill？
还是直接生成图片？
```

这些都应该由商品图总入口判断。

总入口是：

```text
.agents/skills/ecommerce-product-image/SKILL.md
```

可以把它理解成这套项目的负责人。

它负责：

- 接收商品图任务；
- 检查用户提供了什么资料；
- 判断需不需要看竞品；
- 判断有没有评价数据可以分析；
- 汇总所有结果；
- 生成整套图片计划；
- 等用户确认；
- 拆成单图任务；
- 控制后面的生成和审核。

所以运营不需要“调用 Skill”。

运营只需要进商品图入口，后面的专业分工由它自己完成。

## 三个 Skill 的关系，像一个负责人带两个专业角色

### 商品图 Skill：项目负责人

```text
.agents/skills/ecommerce-product-image/SKILL.md
```

它掌握整条主流程。

但它不会自己包办所有分析。

遇到竞品问题，就把任务交给竞品分析；有评价数据，就让评价分析去提炼用户关注点。最后所有结果再回到商品图 Skill，由它统一形成最终方案。

### 竞品分析 Skill：看市场怎么卖

```text
.agents/skills/competitor-analysis/SKILL.md
```

它负责看：

- 竞品第一张图为什么这样做；
- 产品占比有多大；
- 用什么背景和场景；
- 文案先讲什么；
- 主图之间怎么分工；
- 详情页按什么顺序说服用户；
- 哪些可以借鉴；
- 哪些不应该照搬。

它只负责分析。

它不会自己决定 X51C 最终要做哪 6 张主图，也不会分析完马上出图。

### 评价分析 Skill：看用户在意什么

```text
.agents/skills/review-analysis/SKILL.md
```

它负责把用户反馈转成图片机会。

例如：

```text
用户经常问“支持鸿蒙吗”
→ 兼容系统应该主动讲清楚

用户经常误解包装里有鼠标
→ 套装图应该明确说明

用户说连接很快
→ 可以判断它是否值得升级为重点卖点
```

但不是所有评价问题都要做成图片。

物流慢、客服态度、质量缺陷这类问题，不一定应该塞进商品图里解决。

而且这个 Skill 是可选的。

没有评价数据时，项目直接跳过，不应该为了流程完整硬造一个评价分析结果。

## X51C 实际跑起来是什么样

X51C 这次的入口资料主要有：

```text
X51C产品属性.md
实拍图片/
竞品链接
用户反馈关键词
```

商品属性里已经确认了：

- 单蓝牙；
- 带触控板；
- 带手托；
- Type-C 充电；
- 280mAh 电池；
- 支持 Windows、Mac/iOS、Android、HarmonyOS；
- 约 30 分钟自动休眠；
- 12 个月保修。

竞品分析返回的重点是：

- 同类产品常用平板 + 键盘的轻办公场景；
- 前几张图会优先讲触控、便携和系统兼容；
- 详情页通常从场景进入，再讲功能和参数；
- X51C 可以把“触控板 + 手托 + 多系统 + HarmonyOS”讲得更清楚。

评价和用户反馈的价值，不是重新定义产品，而是帮助判断什么更值得讲。

这些信息最后全部回到商品图总入口。

总入口再决定：

```text
主图 1 做整体场景
主图 2 专门讲触控板
主图 3 讲多系统兼容
主图 4 讲手托
主图 5 讲便携场景
主图 6 补参数和信任
```

同时还规划了 10 屏详情图。

也就是说，竞品 Skill 和评价 Skill 都不是最终决策者。

真正决定“整套图片怎么做”的，还是商品图总入口。

## 资料不足时，项目应该停下来，而不是让 AI 自己补

这一点是我现在很重视的。

商品图任务里，缺资料不代表所有任务都停；但依赖这个资料的图片必须停。

例如：

```text
缺少重量
但当前只做触控板图
→ 不影响当前任务

缺少侧面照片
但要做轻薄侧面图
→ 这张图必须暂停

缺少接口照片
但要做 Type-C 接口图
→ 不能让 AI 自己猜接口位置
```

这时候正确动作不是：

```text
根据常识补一个接口
```

而是：

```text
暂停当前图片
↓
告诉用户缺什么
↓
给出补拍要求
↓
用户补照片
↓
继续原来的任务
```

例如可以直接要求：

```text
当前缺少 X51C Type-C 接口的真实照片。
请补拍：

1. 接口完整位置；
2. 机身侧面结构；
3. 光线清楚；
4. 不要遮挡接口。
```

照片质量也一样。

如果图片太暗、角度遮挡、关键结构看不清，项目应该说明为什么不能用，并让用户重新拍。

对这套项目来说，“要求补拍”不是失败，而是正常流程的一部分。

## 两个 Markdown 文件，是项目最重要的状态

这套项目以后是否好排查，很大程度取决于两个文件。

### `商品图片生成计划.md`

它保存整套方案。

里面应该能看到：

```text
商品事实
竞品分析结论
评价分析结论
卖点分级
主图规划
详情图规划
视觉方向
禁止内容
待补资料
```

所以以后发现：

```text
整套图片都不该这样做
```

不要先去改某一张图。

应该先查：

```text
商品图片生成计划.md
```

因为这属于整套方案问题。

### `商品图片单图任务.md`

它保存单张图片怎么执行。

例如 X51C 主图 2 需要写清：

```text
图片目标
核心卖点
标题和副标题
画面构图
使用哪张真实产品图
哪张图负责结构校准
允许改什么
禁止改什么
审核标准
```

所以以后只有某一张图有问题，例如：

```text
触控板太小
标题位置不对
产品占比不合适
```

先查单图任务，而不是推翻整套方案。

## X51C 主图 2 是怎么真正执行的

这张图专门讲触控板。

在整套计划里，它先被定义成：

```text
图片目标：解释为什么选择触控款
核心卖点：减少输入时频繁抬手触屏
主标题：带触控板 操作更顺手
```

到了单图任务，又继续拆成：

```text
主要角度参考：IMG_3292.jpg
结构校准参考：IMG_3281.jpg
画面：45° 产品视角 + 触控板局部强调
允许：增加手指轻触示意
禁止：移动触控板、改变按键、增加不存在的功能
```

然后才交给 GPT Image 2。

这里两张真实图片的作用并不一样。

```text
IMG_3292.jpg
→ 告诉模型主要从什么角度画

IMG_3281.jpg
→ 告诉模型真实结构到底长什么样
```

所以最终输入更接近：

```text
真实角度依据
+
真实结构依据
+
单图执行任务
+
禁止修改内容
↓
GPT Image 2
```

这也是为什么我不再把商品图问题简单理解成 Prompt 写得好不好。

## GPT Image 2 只是最后的生成层

项目架构里，GPT Image 2 很重要，但它的位置其实很靠后。

它接收到的应该是：

```text
已经确认的单图任务
+
真实产品图
+
结构参考图
+
明确的禁止项
```

它的职责是完成画面。

它不应该重新判断：

- 商品到底有什么功能；
- 哪个卖点最重要；
- 这张图是不是应该做；
- 接口应该长在哪里；
- 包装里是不是应该有鼠标。

这些都应该在生成之前解决。

## 当前这套项目跑到了哪里

X51C 现在已经完成：

```text
商品资料整理
✓
竞品分析规则
✓
评价分析规则
✓
整套图片计划
✓
人工确认
✓
单图任务拆解
✓
GPT Image 2 测试生成
✓
```

实际已经生成了 3 张首图方向测试和 1 张触控板卖点图。

还没有完成：

```text
整套主图和详情图全部生成
→
完整人工审核
→
正式上线
→
点击率和转化率对比
→
根据真实数据继续改图
```

所以现在更准确的状态是：

> **项目已经跑通了从资料、Skill 分工、计划确认到受控单图生成的链路，但上线后的数据闭环还没完成。**

## 最后，我现在怎么理解这个项目

以前我会把它看成一个“AI 商品图生成项目”。

现在我更愿意把它看成一条完整的商品图生产线。

```text
运营负责提供真实资料和确认方案
↓
商品图 Skill 负责控制主流程
↓
竞品 Skill 看市场
↓
评价 Skill 看用户
↓
资料闸门决定继续还是补拍
↓
整套计划保存方向
↓
单图任务保存执行细节
↓
GPT Image 2 负责生成
↓
人工审核决定通过还是返回修改
```

这套结构最大的价值，不只是让运营更容易用。

对我自己来说，以后遇到问题也终于知道该往哪里查。

<HighlightBox eyebrow="最后的判断" title="一套好用的 Agent 项目，最好让“怎么使用”和“怎么排查”共用同一张地图">
  运营只需要知道从哪里开始、什么时候确认、什么时候补资料；维护项目的人则继续往下看 Skill、状态文件和生成层。正常流程走得通，异常时也知道该回哪一层，这套项目才算真正清楚。
</HighlightBox>
