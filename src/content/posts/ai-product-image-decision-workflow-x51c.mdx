上一篇文章里，我写过一套 [AI 商品图 Agent 工作流](/articles/ai-product-image-agent-workflow/)：人工提供真实商品资料，Agent 分析竞品，生成图片计划，确认后再出图。

那篇更像是方案设计。

这一次，我真的拿 **X51C 触控蓝牙键盘** 跑了一遍。

项目已经从商品资料整理，走到图片计划定稿、单图任务拆解和 GPT Image 2 测试生成。实际跑下来，我对这件事最深的感受反而不是“AI 出图变强了”，而是：

<HighlightBox eyebrow="这次实践的核心" title="商品图的问题，往往发生在出图之前">
  如果商品事实、竞品分析、卖点优先级和图片任务都没有先决定清楚，模型画得越快，只会越快地产生一批方向不稳定、事实不可靠的图片。
</HighlightBox>

## 一开始的问题：AI 会画，但它不知道为什么这样画

最直接的商品图做法很简单：

```text
商品资料
+
几张参考图
+
一句 Prompt
↓
生成图片
```

这条路径不是不能用。做简单背景替换、抠图、促销小改动时，它甚至很方便。

但做一整套新品主图和详情图时，问题很快就出来了。

模型必须同时猜很多事情：

- 这款商品最重要的卖点是什么；
- 第一张图为什么值得点击；
- 第二张图应该继续讲什么；
- 哪些竞品表达可以借鉴；
- 哪些参数属于我方商品；
- 哪些画面可以调整，哪些产品结构绝对不能动；
- 用户真正会担心什么。

这些其实不是“画图”问题，而是运营判断。

如果把它们全部压进一句 Prompt，相当于让模型一边当商品经理，一边当运营，一边当美工，最后还要自己审核自己。

我后来决定把这件事拆开：

> **先决定画什么，为什么这样画，再让模型负责把它画出来。**

## X51C 的输入，不是一句 Prompt，而是一组可以核对的事实

X51C 是一款带触控板的单蓝牙键盘。

项目开始时，我先整理了一份 `X51C产品属性.md`。里面不是营销文案，而是后面所有判断的事实底座。

已经确认的内容包括：

- 单蓝牙连接，蓝牙名称是 BOW5.0；
- 带触控板和手托；
- Type-C 充电；
- 内置 280mAh 电池；
- 支持 Windows、Mac/iOS、Android、HarmonyOS；
- 约 30 分钟无操作自动休眠；
- 保修期 12 个月。

同时，也有一批当时没有确认的内容：

- 具体尺寸、重量和厚度；
- 实际 SKU 颜色名称；
- 包装清单；
- 产品材质和键帽材质；
- 部分标准角度的实拍图。

这一步看起来很基础，但它后来直接决定了哪些图可以继续做，哪些图必须停下来。

例如，计划里有一张图要讲 Type-C 充电。如果没有真实接口照片，Agent 不能根据“Type-C 接口一般长这样”自己补一个接口位置。

同样，竞品有三模、背光、鼠标套装，也不代表 X51C 可以跟着写。

所以这套流程里，我给事实做了三种状态：

```text
已确认
→ 可以进入图片计划

待补充
→ 可以继续做不依赖它的部分
→ 依赖它的单图任务暂停

禁止推测
→ 再像竞品、再好看，也不能写进去
```

这让我第一次觉得，商品图工作流里的“拒绝生成”其实和“生成图片”一样重要。

## 看竞品，不是为了抄卖点，而是为了看市场怎么表达

竞品分析阶段，我真正关心的不是“别人有什么功能”，而是：

- 第一张图为什么能吸引人；
- 产品主体占多大；
- 触控板是怎么被突出出来的；
- 详情页先讲场景、痛点还是功能；
- 用户看完前几张图，能不能快速理解这是什么商品。

X51C 这次得到的竞品共性很明确：

同类产品经常使用平板、键盘、手机、咖啡和桌面道具组成柔和场景；低饱和粉、绿、白比较常见；后续主图会补键帽近景、触控板局部、轻薄便携、套装和参数。

这些可以借鉴。

但竞品里出现的三模连接、静音分贝、背光、磁吸、鼠标套装，就不能因为“看起来好卖”直接变成 X51C 的卖点。

最终我看到的机会点，不是复制竞品，而是把 X51C 自己已经确认的几个点讲得更清楚：

```text
触控板
+
手托
+
多系统兼容
+
支持 HarmonyOS
+
单蓝牙配对路径清楚
```

这一步的作用，是把“市场怎么说”变成“我们应该怎么说”，中间必须经过一次判断。

## 卖点要排队，不是全部冲进第一张图

刚开始做商品图时，我很容易陷入一个问题：既然这些都是卖点，那就都写上去。

结果就是每张图都很努力，每张图都没有重点。

X51C 这次我把卖点分成了三级。

| 层级 | X51C 实际内容 | 主要作用 |
| --- | --- | --- |
| 一级卖点 | 触控板、多系统兼容、轻薄便携 | 决定用户是否继续看 |
| 二级卖点 | HarmonyOS、手托、圆点键帽、单蓝牙、Type-C | 建立差异和使用理由 |
| 三级卖点 | BOW5.0、280mAh、自动休眠、12 个月保修 | 完成参数和购买确认 |

这不是在给卖点打分，而是在决定它们应该出现在哪里。

例如触控板值得单独占一张主图，因为它能解释“为什么要选这一款”。

而蓝牙名称 BOW5.0 很重要，但更适合出现在连接步骤和参数区域，不适合拿来抢第一张图的注意力。

## 真正改变流程的，是一份中间决策文件

竞品分析完成后，我没有直接让 Agent 出图。

中间先生成了一份：

```text
商品图片生成计划.md
```

这份文件最后规划了：

- 6 张主图；
- 10 屏详情图；
- 一级、二级、三级卖点；
- 每张图的目标；
- 核心文案；
- 画面设计；
- 真实产品图要求；
- 禁止出现的内容；
- 仍然缺少的资料。

例如主图 2 的任务不是“做一张触控板商品图”，而是：

```text
图片目标：解释为什么选择触控款
核心卖点：带触控板，减少频繁抬手触屏
主标题：带触控板 操作更顺手
画面重点：触控板局部放大 + 手指轻触示意
禁止内容：替代鼠标、专业绘图、复杂高级手势
```

这份计划最有价值的地方，是把很多原本只存在脑子里的判断写了出来。

我可以在出图前发现方向错了，也可以直接修改“为什么做这张图”，而不是生成后只盯着背景、字体和颜色反复改 Prompt。

从这一点开始，`商品图片生成计划.md` 不再是草稿，而更像这套工作流的决策中间件。

## 一张图真正执行时，还要再拆一次

计划定稿后，我又生成了 `商品图片单图任务.md`。

整套方案到这里才开始被拆成一张张独立任务。

以主图 2 为例，单图任务进一步固定了：

- 产品使用哪个实拍角度；
- 哪张图负责校准产品结构；
- 允许修改哪些内容；
- 不允许修改哪些内容；
- 文案必须写什么；
- 哪些词绝对不能出现；
- 最后怎么判断这张图合格。

这一步让我觉得很像 RPA。

大的业务流程不会直接写成“一步完成”，而是继续拆到具体动作、输入和校验条件。商品图也一样。

`商品图片生成计划.md` 负责决定整套图片怎么卖商品；`商品图片单图任务.md` 负责告诉模型这一张具体应该怎么执行。

## 为什么触控板图用了两张真实产品图

主图 2 的测试生成里，我没有只给模型一张图。

两张实拍图承担了不同任务：

```text
IMG_3292.jpg
→ 主要角度参考
→ 提供触控板清楚可见的 45° 产品视角

IMG_3281.jpg
→ 严格结构参考
→ 校准键盘布局、触控板位置、按键数量和整体比例
```

然后才叠加单图任务中的画面要求。

也就是说，最终输入不是：

```text
一张图 + 请帮我做得更好看
```

而是：

```text
角度依据
+
结构依据
+
明确的图片任务
+
禁止修改内容
↓
GPT Image 2
```

Prompt 里还专门限制了：不能移动触控板，不能变成笔记本那种居中触控板，不能增加按键，不能删除指示区域，也不能为了画面好看改变产品颜色和外壳比例。

这比单纯写“保持产品一致”具体得多。

因为实际出图时，AI 最容易出问题的地方往往不是背景，而是它觉得某个结构“这样画更合理”。

但商品图不能让模型替真实产品重新设计一遍。

## 当前已经做到哪一步

目前 X51C 这套流程已经完成：

```text
商品资料整理
✓
竞品分析规则
✓
评价分析规则
✓
卖点分级
✓
商品图片生成计划
✓
人工确认
✓
单图任务拆解
✓
GPT Image 2 测试生成
✓
```

实际已经生成了 3 张首图方向测试和 1 张触控板卖点图。

但还没有完成的是：

```text
整套主图和详情图全部生成
→
完整人工审核
→
正式上线
→
点击率 / 转化率对比
→
根据真实数据继续修改图片计划
```

所以现在还不能说这是一套已经验证成功的商品图系统。

更准确地说，它已经把“从商品资料到受控单图生成”这段链路跑通了，后面的上线数据闭环还需要继续做。

## 这次实践让我改了一个看法

以前我会把 AI 商品图的重点放在模型、Prompt、参考图数量和生成参数上。

现在我更关注另外几个问题：

```text
商品事实有没有锁定？
竞品信息有没有被错误借用？
卖点有没有先排优先级？
每张图有没有明确任务？
缺少关键角度时，流程会不会主动停下来？
生成前有没有人确认？
```

这些问题没有答案时，换一个更强的图片模型也只是提高“随机出一张好图”的概率。

而当这些判断被固定下来以后，模型才真正变成一个执行节点。

<HighlightBox eyebrow="最后的结论" title="我不是先让 AI 学会画商品图，而是先让它学会什么时候可以画、应该画什么">
  X51C 这次实践真正沉淀下来的，不是一条万能 Prompt，而是一条可以检查、暂停、修改和继续执行的决策链。图片生成只是最后一步，前面的事实、判断、计划和确认，才决定这张图能不能真正拿去卖商品。
</HighlightBox>
