## 任务拆解

自动化项目最容易失控的地方，是把输入、工具、输出、人工确认和失败处理混在一起。第一步不是抽象，而是把这些节点按顺序摊开。

<Callout>
核心原则：复杂逻辑可以存在，但状态变化和失败边界必须显式。
</Callout>

<Callout title="提示">
先把流程画清楚，再决定是否需要抽函数或拆模块。
</Callout>

## 先判断任务应该走哪条路

很多 Agent 工作流会变复杂，不是因为任务本身复杂，而是因为一开始没有判断任务类型。小改动被套上大流程，会拖慢执行；大任务没有状态文件，又会在中途丢上下文。

所以我更推荐先做一次轻量判断：**能直接改就直接改，需要阶段状态才建立 `TASK.md`，会产生副作用的任务必须加日志、验证和人工确认。**

<AgentWorkflowDecisionTree />

这个决策树的价值，不是让所有任务都变正式，而是防止两种极端：一种是小事复杂化，另一种是大任务只靠对话记忆硬跑。

## 大任务用 TASK.md 管状态

当任务涉及多阶段迁移、多文件整理、知识库重构或规则体系调整时，不要完全依赖对话上下文。更稳的方式是先生成一个临时 `TASK.md`，把它作为本次任务的状态文件。

<Callout title="适用场景">
`TASK.md` 只适合大任务。小改一个函数、删除几行规则、调整少量文案时，不要额外引入这个流程。
</Callout>

### 执行方式

1. 先生成 `TASK.md`
   - 写清目标
   - 写清阶段
   - 写清验收标准

2. 每次执行前先读 `TASK.md`
   - 看当前阶段
   - 看未完成项
   - 看风险点

3. 每完成一个阶段，更新 `TASK.md`
   - 状态改为已完成
   - 补充实际改动
   - 记录遗留问题

4. 阶段完成后再继续下一阶段
   - 不要一次性全迁移
   - 先确认结构和索引规则稳定

    G -- 否 --> C`}
/>

### TASK.md 推荐结构

```md
# 当前任务目标

整理目标、最终效果和验收标准。

## 执行阶段

- 阶段 1：结构与目录规则，状态：未开始
- 阶段 2：索引与命名规则，状态：未开始
- 阶段 3：正文迁移，状态：未开始
- 阶段 4：清理旧内容，状态：未开始

## 当前进度

当前做到哪一步。

## 未完成事项

- 还没处理的内容
- 需要继续验证的内容

## 风险 / 待确认

- 可能影响旧入口的地方
- 需要人工确认的取舍
```

这个文件不是聊天记录合集，而是任务看板。只保留目标、阶段、进度、未完成项、验收标准和风险点。

## 工具边界

工具只做确定的动作，模型负责判断和组织上下文。对于网页自动化、表格写入、文件落盘这类操作，应该让结果可验证。

```python
# keep the flow explicit
result = agent.run(task)
if result.needs_review:
    notify_user(result.summary)
save_run_log(result)
```

## 失败处理

不要吞异常，不要隐藏副作用。逐项处理就逐项保存，整批处理就批量末尾保存，让失败隔离粒度和落盘时机一致。
