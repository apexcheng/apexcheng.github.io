  更稳的做法是：人工先提供真实商品资料，Agent 再分析竞品图片和页面，提炼卖点，落盘 MD 计划文档，和用户对话确认后，再按定稿计划生成主图和详情图。

通用生图工具的问题不是不能出图，而是很难稳定理解电商转化逻辑。它可以生成一张好看的图片，但不一定知道首图为什么要吸睛、详情图为什么要按顺序说服用户，也不一定能保证产品结构、参数、配件和文案都准确。

所以这个方案的重点不是“让 AI 随机画图”，而是把商品图生产流程拆成一套可确认、可落盘、可迭代的 Agent 工作流。

## 方案概览

这套流程里，Agent 不是从零猜商品，也不是直接替代美工。它更像一个“商品图策划助手”：先分析竞品怎么表达，再结合我们自己的真实资料，把主图和详情图设计成一条完整销售链路。

## 一、先明确职责边界

这套方案最重要的边界是：**事实由人工提供，分析和创作由 Agent 完成。**

如果让 Agent 自己猜商品参数、功能、配件和材质，后面的图片再好看也可能不能用。尤其是商品图这种场景，产品外观、接口、颜色、套装、卖点和平台合规都不能乱来。

  <FeatureCard title="人工提供真实资料" meta="INPUT" description="人工只负责把商品事实说清楚，避免 Agent 根据常识瞎猜。">
    <ul>
      <li>商品名称、型号、品牌、类目</li>
      <li>真实产品图、包装图、细节图、套装图</li>
      <li>尺寸、重量、材质、功能、接口等参数</li>
      <li>SKU、价格定位、已知卖点和禁止夸大内容</li>
      <li>目标平台和 3 到 5 个竞品链接</li>
    </ul>
  </FeatureCard>
  <FeatureCard title="Agent 负责后续流程" meta="WORKFLOW" description="Agent 从竞品链接开始介入，负责分析、策划、落盘和出图。">
    <ul>
      <li>打开竞品页面，读取标题、价格、规格和卖点</li>
      <li>截图并对主图、详情图做 OCR 分析</li>
      <li>提炼一级、二级、三级卖点</li>
      <li>设计主图和详情图结构，生成 MD 计划文档</li>
      <li>根据用户反馈修订，定稿后生成图片</li>
    </ul>
  </FeatureCard>
</VisualGrid>

这一步看起来像前置准备，但其实决定了整套流程能不能稳定。如果输入资料不真实、不完整，Agent 很容易生成“视觉上很像商品图，但商品信息不可靠”的图片。

## 二、为什么必须分析竞品图片

电商页面里很多关键信息不是写在商品详情文字里，而是写在主图、卖点图、对比图和场景图里。只读页面标题和参数，通常看不出竞品真正想打什么。

Agent 分析竞品时，不能只看“这个竞品有什么优势”，还要回答几个更具体的问题：

- 竞品首图为什么吸睛？
- 它把产品放多大？背景是什么风格？
- 它第一眼想让用户看到哪句话？
- 它详情页是先讲痛点，还是先讲功能？
- 它有没有通过对比图、场景图、参数图建立信任？
- 它没有讲清楚的地方，能不能变成我们的机会点？

<Callout title="重点">
竞品分析不是为了照抄竞品，而是为了看清同类商品已经在讲什么、用户可能关心什么，以及我们应该避开什么表达方式。
</Callout>

### OCR 要提取什么

| 图片类型 | Agent 需要分析的内容 |
| --- | --- |
| 首图 | 核心文案、产品占比、背景、吸睛方式、点击理由 |
| 主图二 | 最大卖点的表达方式，以及是否能一眼看懂 |
| 对比图 | 和普通款、旧款或竞品如何做差异化表达 |
| 场景图 | 使用场景、目标人群、画面氛围和代入感 |
| 详情图 | 痛点顺序、卖点顺序、参数说明和说服路径 |
| 套装图 | 配件展示方式，以及是否突出组合价值 |
| 属性图 | 参数是否清晰，信息密度是否合适 |

最终每个竞品都应该形成一段结构化分析，而不是只输出一句“这个竞品主打性价比”。

```md
## 竞品 A 分析

- 链接：
- 标题：
- 价格：
- 主推卖点：
- 首图特点：
- 产品占比：
- 背景风格：
- OCR 提取文案：
- 详情页结构：
- 可借鉴点：
- 不建议照搬点：
- 我们的机会点：
```

## 三、卖点要分级，不是全部堆到图上

很多商品图做不好，不是因为没有卖点，而是把所有卖点都放在同一层级。主图一、主图二、详情图、参数图承担的任务不一样，卖点也应该分层。

| 卖点等级 | 作用 | 适合放哪里 |
| --- | --- | --- |
| 一级卖点 | 决定用户是否点击和继续看 | 主图一、主图二 |
| 二级卖点 | 支撑用户继续了解和比较 | 主图三、主图四、详情页前半段 |
| 三级卖点 | 补充信任、参数和购买确认 | 详情页后半段、属性图、套装图 |

比如一个键鼠类商品，一级卖点可能是三模连接、长续航、静音按键；二级卖点可能是多设备切换、轻薄便携、高颜值配色；三级卖点则是尺寸、重量、兼容系统和包装清单。

<Callout title="判断标准">
一级卖点必须适合被用户一眼看懂。需要解释很久的内容，可以放进详情页，不要硬塞进首图。
</Callout>

## 四、主图和详情图要分工

主图的核心目标是让用户点进来，并在前几张图里快速建立商品优势。详情图的核心目标是让用户理解为什么需要、为什么选我们、买到什么，以及是否放心。

### 主图结构

| 图片 | 目标 | 内容方向 |
| --- | --- | --- |
| 主图一 | 吸睛点击 | 产品大图 + 核心视觉钩子 + 最强利益点 |
| 主图二 | 讲最大卖点 | 用视觉方式表达核心功能或核心优势 |
| 主图三 | 建立差异 | 与普通款、旧款、竞品或用户痛点做对比 |
| 主图四 | 展示场景 | 办公、学习、家用、游戏、出差等场景 |
| 主图五 | 补充信任 | 套装、细节、参数、材质、兼容性 |

主图一尤其重要。它不是单纯把产品放大，也不是文案越多越好，而是要解决一个问题：用户为什么要点进来？

### 详情图结构

详情图不是简单堆卖点，而是一个说服链路：先让用户觉得“我需要这个”，再告诉用户“这个产品怎么解决问题”，最后补充“买到什么”和“是否放心”。

## 五、MD 计划文档是核心中间产物

这套流程里，Agent 不能分析完竞品就直接出图。中间必须落盘一份 MD 计划文档。

这个文档有三个作用：

1. 把竞品分析、OCR 结果、卖点提炼和图片结构沉淀下来；
2. 让用户能在出图前确认方向，避免生成后大改；
3. 让最终图片生成有明确约束，而不是靠一句模糊提示词。

建议文件名可以直接叫：

```text
商品图片生成计划.md
```

## 六、对话定稿比一次性生成更重要

商品图方案不应该由 Agent 一次性拍板。更好的方式是让 Agent 先给出初版计划，再围绕关键问题和用户对话确认。

<VisualGrid columns={3}>
  <FeatureCard title="确认首图方向" meta="主图" description="首图可以走强卖点、场景氛围、颜值吸睛或价格利益方向。" />
  <FeatureCard title="确认最大卖点" meta="卖点" description="不要同时推太多核心卖点，主图二最好只表达一个最强优势。" />
  <FeatureCard title="确认表达边界" meta="合规" description="哪些参数不能写满，哪些效果不能夸大，哪些竞品信息不能出现。" />
</VisualGrid>

文档版本可以这样流转：

```text
初版计划.md
↓
用户反馈
↓
修订版计划.md
↓
用户确认
↓
最终定稿计划.md
↓
生成图片
```

这样做虽然多了一步，但能明显减少返工。因为很多分歧应该在“图片脚本阶段”解决，而不是等图片已经生成后再推倒重来。

## 七、最终出图必须受控

最终生成图片时，Agent 不应该再自由发挥。每张图都应该从最终 MD 计划里拆成独立生成任务。

```md
# 主图一生成任务

## 图片目标

提升点击率，第一眼突出商品核心吸引点。

## 画面设计

- 产品位于画面中心；
- 产品占画面 60% 以上；
- 背景使用简洁科技感、生活场景或品牌风格背景；
- 核心文案放在视觉焦点区域；
- 辅助卖点弱化展示。

## 图片文案

主标题：
副标题：
辅助文案：

## 产品要求

- 必须使用真实产品图；
- 不改变产品外观；
- 不新增不存在的配件；
- 不改变颜色；
- LOGO 不变形。

## 禁止内容

- 不使用“全网第一”“最强”“顶级”等绝对化词语；
- 不虚构参数；
- 不出现竞品品牌；
- 不出现商品没有的功能；
- 不出现错误配件。
```

这里的重点是“受控”。商品图不是艺术图，不能为了视觉效果牺牲商品真实性。

## 八、人工审核和上线测试不能省

AI 生成图片后，最后一关仍然应该是人工审核。尤其要检查产品外观、颜色、结构、接口、配件、参数和文案是否符合真实商品。

| 审核项 | 说明 |
| --- | --- |
| 产品外观 | 是否和真实产品一致，是否出现变形或错误结构 |
| 产品颜色 | 是否和真实 SKU 一致，是否误导用户 |
| 配件套装 | 是否新增了不存在的配件，或者漏掉关键配件 |
| 文案准确性 | 是否有虚假参数、夸大效果或绝对化表达 |
| 平台规则 | 是否符合目标平台主图和详情图要求 |
| 转化逻辑 | 图片顺序是否能让用户逐步理解和信任 |

上线后还要看数据，尤其是主图一点击率、商品转化率、收藏加购率、详情页跳失情况、用户咨询问题和差评反馈。

<HighlightBox eyebrow="CHECK" title="最终判断标准">
  商品图是否成功，不只看图片好不好看，而要看它有没有让用户更愿意点击、更容易理解卖点、更少产生疑问，并且没有偏离商品真实情况。
</HighlightBox>

## 九、这套方案适合做成固定 Agent 流程

如果要把它产品化或工具化，可以把流程固化成几个阶段：

<DecisionFlow
  title="可固化的 Agent 执行阶段"
  items={[
    {
      label: '阶段 1',
      title: '读取人工资料',
      description: '只接受用户提供的商品事实，不主动编造参数、功能和配件。',
    },
    {
      label: '阶段 2',
      title: '分析竞品链接',
      description: '打开页面，截图主图和详情图，并通过 OCR 提取图片文案。',
    },
    {
      label: '阶段 3',
      title: '生成图片计划',
      description: '把竞品分析、卖点分级、主图结构和详情图结构落盘到 MD 文档。',
    },
    {
      label: '阶段 4',
      title: '对话确认定稿',
      description: '围绕首图方向、最大卖点、文案风格和禁止项与用户确认。',
    },
    {
      label: '阶段 5',
      title: '按计划出图',
      description: '根据最终计划逐张生成图片，再交给人工审核和上线测试。',
    },
  ]}
/>

这个流程真正有价值的地方，不是“AI 会画图”，而是把商品图从一件靠感觉的事情，变成一套能沉淀、能复用、能不断迭代的工作流。

## 总结

AI 商品图工作流的核心，不是让 Agent 替人拍脑袋，而是把人的商品事实、竞品的视觉表达、平台的转化逻辑和 AI 的生成能力接起来。

最关键的三个原则：

1. 人工输入事实，Agent 负责分析和创作；
2. 先竞品分析和 MD 计划，后对话定稿和出图；
3. 图片必须服务点击、转化和真实表达，而不是只追求好看。

<HighlightBox eyebrow="一句话" title="先计划，后出图">
  只要商品资料、竞品 OCR、卖点分级、图片计划和用户确认这几个节点稳定下来，AI 生成商品主图和详情图才有机会从“碰运气”变成“可复用流程”。
</HighlightBox>
