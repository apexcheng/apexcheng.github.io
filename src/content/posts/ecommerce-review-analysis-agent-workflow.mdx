这篇文章来自我在 Mac mini 上长期运行的 `review-analyst` 项目。它读取钉钉 AI 表格里的评价数据，生成日报或周期对比报告，再把结果发布成钉钉文档并发送到群里。

公开版本去掉了真实店铺、订单号、评价原文、手机号、Webhook 地址、表格地址和内部配置，只保留系统怎么判断、怎么执行、怎么避免误写状态。

## 它不是一个“评价总结 Prompt”

最开始看，这个任务很像一句简单需求：

```text
读取今天的评价，分析有哪些问题，然后发到群里。
```

但真正执行时，至少会遇到这些问题：

- 首次评价和追加评价分布在两张表，字段并不相同。
- 情绪、问题类型和责任归因已经由上游字段给出，Agent 不能再凭文案重新判断。
- 日报只应该处理还没有通知过的评价，周期对比则不能受通知状态影响。
- 群消息发送失败时，不能把这些记录提前标记为“已通知”。
- 两张表的 `recordId` 不能混在一起回写，否则会改错数据。

所以我没有把它做成“模型直接读表，然后自由发挥”。我把它拆成了一个事实层、一个表达层和一个交付层。

这个分工最重要的价值，是把“确定事实”和“组织语言”分开。Python 不负责写漂亮报告，Agent 也不能绕开固定口径直接改业务状态。

## 两张表先变成一种评价

项目默认同时读取：

| 来源 | 原始内容字段 | 进入统计周期的时间 | 统一后的来源标记 |
| --- | --- | --- | --- |
| 首次评价 | `评价内容` | `评价时间` | `首次评价` |
| 追加评价 | `追加评价内容` | `首次评价时间` | `追加评价` |

追加评价最容易产生误解。表里虽然有“追加评价时间”，但当前项目不会拿它决定统计周期，而是使用“首次评价时间”。这个口径先由脚本固定，再统一输出为：

```text
review_source      评价来自哪张表
review_time        实际参与筛选和统计的时间
first_review_time  首次评价时间
append_review_time 追加评价时间，仅供参考
评价内容           两张表统一后的文本字段
```

两张表合并后，还会按下面三个字段去重：

```text
订单号 + 商品ID + 评价内容
```

这里的目的不是判断哪条记录更新，也不是删除源表数据，只是避免同一条评价被重复统计。

## 为什么一定要先生成事实包

事实包是这套系统的中间层，也是 Agent 唯一允许使用的数据来源。

脚本会把 JSON 放在两个明确标记之间：

```text
REVIEW_REPORT_CONTEXT_JSON_START
{ 本次任务的统一事实 }
REVIEW_REPORT_CONTEXT_JSON_END
```

这样做可以挡住三类常见问题：

1. **调试日志混入报告。** Agent 只能读取标记之间的 JSON，stderr 和工具日志不能当成业务事实。
2. **模型补充不存在的原因。** 报告中不能擅自增加订单、批次、质检或供应链信息。
3. **统计和表达互相污染。** Python 先完成数量、占比、排行和明细，Agent 只负责让读者看懂。

<HighlightBox eyebrow="SOURCE OF TRUTH" title="报告写得再好，也不能越过事实包">
  情绪倾向、问题类型和责任归因都服从表格已有字段。Agent 可以概括代表关键词，但不能根据评价文案重新分类，更不能为了让报告更完整而补一个看似合理的原因。
</HighlightBox>

这里有一个很关键的细节：日报里的“非正面评价”固定等于中性加负面；真正放进“重点差评内容”的只有负面。`unknown` 表示上游情绪分析还没完成，本次既不统计，也不回写，让它保留原状态等待下一次处理。

## 发送成功，才允许回写

这部分是整个项目最重要的可靠性设计。

如果系统先把表格记录标记为“已通知”，然后群消息发送失败，那么这些评价下一次也不会再被查询到。业务看到的是“没有收到报告”，表格记录却显示“已经处理”，问题会被静默丢失。

<ReviewAnalysisDeliveryGate />

对应的核心逻辑可以压缩成下面几行：

```python
if webhook_success:
    writeback(current_fact_package_record_ids)
else:
    keep_original_status()
```

回写阶段还会进入一个更窄的 `writeback-only` 模式：

- 只能使用本次事实包生成的 record ids。
- 禁止重新查询评价记录。
- 禁止重新生成一份 daily 事实包。
- 必须按来源表分组回写，不能混用两张表的 recordId。

如果报告已经发送成功，但回写失败，系统不会装作任务完成，而是追加发送一条风险提示：这批评价可能在下一次日报里重复出现。

## 我故意不让 Agent 做的事

真正能长期运行的 Agent，不只是“能做什么”，还要把“不允许做什么”写清楚。

| 禁止动作 | 为什么 |
| --- | --- |
| 根据评价文案重新判断情绪 | 会覆盖上游统一口径，导致同一条评价在不同报告里分类不一致 |
| 把中性评价混入重点差评 | 日报关注口径和重点差评口径不同，混用会放大问题 |
| 使用事实包外信息解释原因 | 容易把推测写成事实，误导运营后续动作 |
| 发送失败仍回写状态 | 会让未送达的评价从后续日报中消失 |
| 回写时重新查询数据 | 当前报告和回写对象可能不再对应，无法保证一次任务的一致性 |
| 把两张表的 recordId 混用 | 同一个 id 只能在自己的来源表里更新 |

这些限制看起来让 Agent 变“笨”了，但它们实际上提高了系统的可信度。模型可以把报告写得更清楚，却不能绕过固定口径和状态控制。

## 这个项目现在解决了什么

以前，评价分析容易停留在一份临时总结：看完就结束，下一次又从头整理。

现在它形成了一条持续工作的反馈链：

```text
评价进入表格
→ 统一口径和状态
→ 生成日报或周期对比
→ 发布在线文档并发送到群
→ 成功后回写处理状态
→ 下一次只处理真正的新数据
```

它并没有替运营直接决定“产品一定要怎么改”。它做的是把散落在评价里的信号稳定送到运营面前，并明确告诉大家：本次看到了什么、哪些值得关注、哪些只是样本有限、哪些状态已经完成。

<HighlightBox eyebrow="最终判断" title="评价分析 Agent 的价值，不是会写报告，而是让每条评价都有可追踪的去向">
  真正困难的部分不是 Prompt，而是数据口径、事实边界、发送结果、状态回写和失败兜底。只有这些环节连起来，评价分析才从一次性的 AI 总结，变成可以长期运行的业务系统。
</HighlightBox>
