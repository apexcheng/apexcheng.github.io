# 文章视觉系统

本文件是正式文章的视觉设计总规则。它解决的问题不是“文章套哪个模板”，而是让 Agent 在写文章前先思考：这篇文章最适合被设计成什么样的视觉表达。

核心目标：**每篇文章都要像被设计过，而不是写完文字后再套骨架。**

---

## 最高原则

```text
先设计视觉表达，再写 MDX 正文。
```

Agent 写正式文章、重写文章或做文章可视化时，必须先判断文章的信息关系，再决定视觉结构。

不要把功能矩阵、报告看板、决策树当成固定模板套用。它们是视觉风格样板，不是文章骨架。

应该学习的是它们的共性：

- 信息分层清楚。
- 文字被拆成可扫读的信息块。
- 标签、编号、状态、流程线都有含义。
- 视觉块像产品界面，而不是普通 Markdown 装饰。
- 低饱和背景、轻边框、圆角、轻阴影、高亮点缀形成统一质感。
- 每个视觉块都服务文章逻辑，不做无意义装饰。

---

## 标准工作流

写文章前按这个顺序执行：

```text
1. 判断文章类型
   ↓
2. 提炼核心信息关系
   ↓
3. 设计文章视觉方案
   ↓
4. 选择或组合视觉组件
   ↓
5. 编写 MDX 正文
   ↓
6. 审查视觉密度和风格一致性
```

禁止直接从长段落进入 MDX。

---

## 视觉方案必须先写出来

长文、新文章、文章重构、复杂排版和可视化改造任务，写 MDX 前必须先给出轻量视觉方案。

建议格式：

```md
## 文章视觉方案

### 文章类型
流程类 / 决策类 / 系统说明类 / 报告复盘类 / 教程类 / 生活记录类

### 核心信息关系
这篇文章主要表达：流程 / 判断 / 对比 / 分层 / 风险 / 行动 / 结论。

### 推荐视觉结构
1. 开头：使用什么总览视觉块，解决读者什么问题。
2. 第一节：使用什么正文核心视觉块，表达什么关系。
3. 第二节：使用什么正文核心视觉块，表达什么关系。
4. 第三节：是否需要补充对比、风险、分层或流程。
5. 结尾：使用什么行动 / 总结视觉块。

### 不适合的表达
说明为什么不只用 Mermaid、普通表格、普通卡片或固定模板。

### 风格参考
说明参考功能矩阵、报告看板、决策树中的哪些视觉气质，而不是复制哪个骨架。
```

小修小补可以不完整输出视觉方案，但仍要遵守“视觉服务内容”的原则。

---

## 每篇正式文章的基础结构

正式文章默认至少包含三类视觉模块：

| 位置 | 模块 | 作用 |
| --- | --- | --- |
| 开头 | 总览视觉块 | 让读者 10 秒内知道文章在讲什么、结论是什么、怎么读 |
| 正文 | 核心视觉块 | 把流程、判断、对比、分层、风险或系统关系可视化 |
| 结尾 | 行动 / 总结视觉块 | 把文章落到判断标准、行动清单或最终结论 |

推荐结构：

```text
文章标题
↓
一句话问题 / 结论
↓
文章总览视觉块
↓
正文短段落 + 核心视觉块
↓
正文短段落 + 核心视觉块
↓
必要细节卡片 / 表格 / 图表
↓
结尾行动清单 / 判断标准 / 总结面板
```

视觉模块数量建议：

| 文章长度 | 推荐视觉密度 |
| --- | --- |
| 800 字以内 | 1 个总览块 + 1 个结论块 |
| 800-1500 字 | 1 个总览块 + 1-2 个核心视觉块 + 1 个结论块 |
| 1500-2500 字 | 1 个总览块 + 2-4 个核心视觉块 + 1 个结论块 |
| 2500 字以上 | 分章节处理，每个大章节至少有一个视觉锚点 |

## 先判断文章应该被看成什么图

Agent 不要先问“套哪个模板”，而要先问：**这篇文章最适合被看成什么图？**

| 文章主题 | 更适合被看成 |
| --- | --- |
| Agent 工作流 | 一张决策 + 执行系统图 |
| RPA 实际流程 | 一条任务从输入到输出的自动化流水线 |
| 商品图方案 | 一条从资料、竞品、卖点到图片产出的生产线 |
| 博客项目结构 | 一张文件、组件、内容、部署的模块地图 |
| GitHub Pages 部署 | 一条构建发布流水线 |
| 审查报告 | 一个状态看板和风险分层报告 |
| 工具选型 | 一棵选择路径清晰的决策树 |
| 生活经验 | 一张情况、判断、边界、行动的记录卡 |
| 教程文章 | 一张学习路线图和步骤检查清单 |

## 文章类型与视觉表达

### 1. 决策类文章

适合：要不要做、什么时候用 A / B、需求如何分流、工具怎么选。

优先视觉：

- 决策路径。
- 分支卡片。
- 风险等级条。
- 推荐动作面板。
- 最终结论面板。

设计方式：

```text
判断入口
├─ 情况 A：直接执行
├─ 情况 B：先确认
├─ 情况 C：不要做
└─ 情况 D：放入 TODO 或人工处理
```

写作要求：

- 判断问题要能回答“是 / 否”。
- 判断层级通常控制在 3 到 4 层。
- 每条路径必须落到明确动作。
- 不要把模糊观点塞进决策树。

适合文章：Agent 任务分流、RPA 是否值得做、普通版和企业版怎么选、功能要不要加入博客。

---

### 2. 流程类文章

适合：自动化流程、部署流程、数据同步流程、商品图生成流程、客服售后流程。

优先视觉：

- 横向流程轨道。
- Stepper 步骤卡。
- Pipeline 输入输出链路。
- 异常分支。
- 多角色泳道。

设计方式：

```text
输入资料
  ↓
Agent 分析
  ↓
生成计划
  ↓
用户确认
  ↓
执行任务
  ↓
审查结果
```

每个节点尽量包含：

```text
谁负责 / 输入什么 / 产出什么 / 失败怎么办
```

写作要求：

- 普通线性步骤不要默认用 Mermaid。
- 如果步骤有责任、输入、产出，优先做成流程卡片。
- 如果存在回退、异常、分支，再考虑 Mermaid 或决策树。

---

### 3. 系统说明类文章

适合：项目结构、代码怎么运行、Agent 系统设计、RPA 项目模块、数据处理系统。

优先视觉：

- 架构分层面板。
- 能力矩阵。
- 文件地图。
- 数据流图。
- 能力边界卡片。

设计方式：

```text
用户输入
  ↓
Agent 层
  ↓
规则层
  ↓
执行层
  ↓
输出层
```

写作要求：

- 先解释系统分几层，再解释每层做什么。
- 文件结构不要只贴目录树，要说明每类文件的职责。
- 能力矩阵只放并列能力，不放流程步骤。

---

### 4. 报告 / 复盘类文章

适合：审查报告、功能上线复盘、体验评估、项目阶段总结、数据分析。

优先视觉：

- 报告看板。
- 指标卡片。
- 风险分层。
- 问题列表面板。
- 下一步行动栈。

设计方式：

```text
当前结论：基本可用，但还有 3 个问题

状态：
- 完成度
- 风险点
- 体验问题
- 下一步
```

写作要求：

- 顶部指标要短，例如 `A-`、`4 层`、`2 处`、`待验证`。
- 风险分层不要超过 5 层。
- 每个问题都要给出状态或下一步动作。

---

### 5. 教程类文章

适合：认识 Agent、AI 工作流程、Prompt、Astro、MDX、影刀 RPA 入门。

优先视觉：

- 学习路线图。
- 步骤卡。
- 概念卡片。
- 常见错误面板。
- 最终检查清单。

设计方式：

```text
你需要先知道什么
  ↓
第一步做什么
  ↓
第二步做什么
  ↓
哪里容易错
  ↓
怎么判断成功
```

写作要求：

- 每一节回答一个学习问题。
- 概念卡片只解释一个概念。
- 结尾必须有“学完后能做什么”或“检查清单”。

---

### 6. 生活记录类文章

适合：健康记录、消费选择、家装方案、工具体验、个人观察。

优先视觉：

- 情况卡片。
- 个人判断面板。
- 边界提醒卡。
- 前后对比。
- 轻量行动清单。

设计方式：

```text
我的情况
↓
我观察到的点
↓
我自己的判断
↓
我会怎么做
↓
注意边界
```

写作要求：

- 视觉风格要更轻，不要像技术后台。
- 少用强科技感和复杂架构图。
- 多用柔和背景、短句、提醒卡和记录清单。
- 涉及健康、法律、金融等内容要明确个人记录和专业边界。

## 视觉组件选择表

| 内容关系 | 优先视觉方式 |
| --- | --- |
| 文章总览 | VisualHero / Bento 总览 |
| 核心结论 | InsightPanel / Verdict Panel |
| 做事步骤 | FlowRail / Stepper |
| 判断条件 | DecisionPath / DecisionTree |
| 功能模块 | CapabilityMatrix / FeatureGrid |
| 数据状态 | ReportBoard / MetricGrid |
| 方案比较 | CompareBoard / 双栏对比 |
| 问题风险 | RiskStack / WarningBoard |
| 文件结构 | FileMap / LayerStack |
| 多角色协作 | RoleSwimlane |
| 前后变化 | BeforeAfter |
| 操作教程 | ActionChecklist |
| 场景拆解 | ScenarioGrid |
| 业务链路 | Pipeline / ProblemChain |

没有现成组件时，可以使用现有 MDX 组件组合，不要为了单篇普通段落过度新增组件。

## 统一视觉风格

### 背景

- 使用低饱和背景。
- 可以用浅渐变、暖白、浅灰、淡蓝灰、米白。
- 避免强烈纯色大背景。
- 避免一篇文章里出现多套完全不同视觉语言。

### 卡片

- 圆角要统一。
- 使用轻边框和轻阴影。
- 留白要充足。
- 每张卡片只表达一个重点。
- 卡片内避免长段落。

### 标签

标签必须表达信息，不是装饰。

常用标签：

```text
输入 / 判断 / 执行 / 风险 / 建议 / 边界 / 产出 / 人工确认 / 自动处理 / 不建议
```

### 颜色语义

| 颜色感觉 | 用途 |
| --- | --- |
| 绿色 | 正常、推荐、可执行、通过 |
| 蓝色 | 信息、系统、流程、说明 |
| 黄色 | 注意、待确认、轻风险 |
| 红色 | 风险、失败、禁止、不建议 |
| 紫色 | Agent、智能分析、策略判断 |
| 灰色 | 背景、辅助信息、未激活状态 |

同一篇文章内颜色语义要一致。

### 字体层级

```text
模块标题：说明这一块在讲什么
卡片标题：说明这个信息点是什么
正文短句：解释必要背景
标签文字：状态、类型、动作或风险
```

视觉块里尽量使用短句，不把正文整段塞进去。

## 三个视觉样板的正确用法

### 功能矩阵

学习它的：

- 并列模块感。
- 能力拆解方式。
- 图标 / 标签 / 简短说明组合。
- 一眼看懂系统能做什么。

不要复制它的：

- 固定卡片数量。
- 固定字段顺序。
- 固定文案结构。

适合：系统能力、模块地图、Agent 职责、项目结构。

---

### 报告看板

学习它的：

- 顶部状态指标。
- 分层防线。
- 风险和状态标签。
- 像验收报告一样的阅读体验。

不要复制它的：

- 固定 L1-L4 层数。
- 固定指标名。
- 固定报告口径。

适合：审查报告、上线验收、稳定性分析、风险控制。

---

### 决策树

学习它的：

- 深色系统感。
- 判断路径。
- 起点、分支、结果之间的关系。
- 侧边触发信号和兜底说明。

不要复制它的：

- 固定判断问题。
- 固定分支数量。
- 固定结论类型。

适合：工具选择、需求分流、自动化是否值得做、方案取舍。

---

## 不允许的问题

- 只是把文字塞进现成卡片。
- 每篇文章都套同一个骨架。
- Mermaid 太小、太密、读不清。
- 大量普通卡片重复堆叠，没有节奏变化。
- 视觉块只有装饰，没有表达流程、判断、对比、风险或结论。
- 表格里塞长段解释。
- 一个视觉块同时表达太多关系。
- 为了好看新增复杂组件，但文章信息并不需要。
- 改文章时只替换局部 HTML，却没有重新设计阅读路径。

---

## 写作时的自检问题

写 MDX 前问：

1. 这篇文章最适合被看成什么图？
2. 读者 10 秒内能不能知道文章讲什么？
3. 哪一段文字最应该被视觉化？
4. 这个视觉块表达的是流程、判断、对比、分层、风险还是结论？
5. 能不能把长段拆成卡片、标签、步骤或状态？
6. 是否只是套了功能矩阵、报告看板或决策树？
7. 这个视觉块在移动端是否还能读？
8. 结尾有没有给出清晰行动、判断标准或总结？

---

## 发布前视觉审查清单

正式文章发布或提交前，检查：

- [ ] 是否先做了视觉方案？
- [ ] 是否有开头总览视觉块？
- [ ] 是否有正文核心视觉块？
- [ ] 是否有结尾行动 / 总结视觉块？
- [ ] 是否根据文章内容做了专属表达？
- [ ] 是否没有把三个视觉样板当固定骨架套用？
- [ ] 是否把长文字拆成了可扫读结构？
- [ ] 是否避免了过多普通卡片？
- [ ] 是否避免了过小、过密、读不清的 Mermaid？
- [ ] 标签、编号、颜色是否有明确含义？
- [ ] 视觉模块是否服务内容，而不是装饰？
- [ ] 移动端是否不会严重挤压变形？

## 推荐落地方式

优先顺序：

1. 能用现有 MDX 组件组合解决，就不要新增组件。
2. 同类文章会重复使用的视觉结构，再沉淀为 Astro 组件。
3. 单篇文章需要强视觉效果时，可以做文章专属组件，但必须说明为什么值得。
4. 视觉实验文章只作为样板间，不要直接搬进正式文章。

写正式文章时，最终目标不是“用了多少组件”，而是：

```text
读者更快看懂文章的结构、判断、风险和下一步。
```
