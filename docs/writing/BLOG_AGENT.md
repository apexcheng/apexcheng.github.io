# 博客写作规则

本文件是本项目的文章写作、审查、发布主规则。处理博客文章内容时读取。

## 核心定位

1. 本站是个人博客，不是官方博客，也不是严格工程文档站。
2. Agent 的职责是整理作者想法，让文章更顺、更适合发布，不要过度干预作者观点。
3. 默认文字风格是专业风格；只有用户明确要求普通读者、大众阅读、口语一点时，才使用大众风格。
4. 如果 Agent 判断主题更适合大众风格，先提醒用户确认，不要直接改风格。

## 内容来源

1. 正式文章目录是 `src/content/posts/`。
2. 视觉实验和 Demo 文章目录是 `src/content/posts/visual-lab/`。
3. 新文章默认复制 `templates/post.mdx` 后填写，优先使用 `.mdx`。
4. 新增或迁移文章后，同步更新 `docs/writing/CONTENT_INDEX.md`。
5. 需要参考旧文章时，先看 `docs/writing/CONTENT_INDEX.md`，只打开 1 到 3 篇最相关正文。
6. 正式文章写作、重写或可视化改造前，先读取 `docs/ARTICLE_VISUAL_SYSTEM.md`，先做文章视觉方案，再写 MDX 正文。
7. 需要参考视觉效果时，读取 `docs/writing/VISUAL_LAB.md` 和 `src/content/posts/visual-lab/`。

## Frontmatter

文章必须包含模板中的 frontmatter 字段。重点规则：

- `date` 使用带时区的完整时间：`YYYY-MM-DDTHH:mm:ss+08:00`。
- `updated` 可选；存在时也使用同样格式。前端只显示日期，时分秒用于同一天文章的精准排序。
- `category` 只能是 `生活`、`实践`、`教程`、`视觉实验室`。
- 新文章默认保持 `draft: false`、`featured: false`、`private: false`。
- 只有用户明确说“草稿”“暂不发布”“仅存仓库”“先不公开”等同等意思时，才设置为 `draft: true`。
- 只有用户明确说“私密”“不进公开列表”“不要公开进入站点”等同等意思时，才设置为 `private: true`。
- `private: true` 只表示不公开进入站点页面和索引，不是密码保护。

## 分类选择

| 分类 | 使用场景 |
| --- | --- |
| `生活` | 健康记录、消费选择、装修、工作感受、日常观察等生活经验 |
| `实践` | 实际做过的项目、工具改造、技术踩坑、博客项目、Agent 工作流、RPA 自动化、数据处理 |
| `教程` | 面向系统学习的文章和系列内容，例如认识 Agent、AI 工作流程、Prompt、Python、Astro |
| `视觉实验室` | 视觉组件、交互效果、页面样式、代码块、图表和文章展示方式等演示内容 |

新文章默认分类可用 `实践`；如果是生活经验，用 `生活`；如果是系统教学或系列内容，用 `教程`。不再使用旧分类：`AI Agent`、`影刀RPA`、`Skill`、`数据处理`。

## 写作流程

长文、新文章或结构不清楚的改写任务，先做轻量 brief：

1. 主题和一句话结论。
2. 目标读者和文字风格。
3. 文章类型：生活经验、技术说明、工作流复盘、RPA 经验、数据处理案例、系统教程、对比决策、清单型笔记等。
4. 素材清单：已有事实、可引用旧文、需要外部资料、需要用户确认的内容。
5. 推荐结构：结论、概览、流程 / 图表、模块拆解、关键细节、检查清单。
6. 视觉方案：这篇文章最适合被看成什么图，开头、正文、结尾分别用什么视觉块。

小修小补可以跳过完整 brief，但不能违背 `docs/ARTICLE_VISUAL_SYSTEM.md` 的核心原则。

## 视觉写作流程

正式文章不能只把文字塞进模板。写 MDX 前先判断文章的信息关系：流程、判断、对比、分层、风险、行动或结论。

默认要求：

1. 开头有总览视觉块，让读者快速理解文章问题和结论。
2. 正文至少有一个核心视觉块，用来表达流程、判断、对比、分层、风险或系统关系。
3. 结尾有行动 / 总结视觉块，把文章落到判断标准、行动清单或最终结论。
4. 功能矩阵、报告看板、决策树只作为视觉风格参考，不作为固定骨架。
5. 每篇文章要根据内容设计专属视觉表达，避免每篇都长得一样。

具体规则以 `docs/ARTICLE_VISUAL_SYSTEM.md` 为准。

## 审查规则

发布或提交前检查：

1. frontmatter 字段完整，分类只使用固定分类之一。
2. 新文章默认公开；只有用户明确要求草稿或私密，才使用 `draft: true` 或 `private: true`。
3. 没有账号、内部资料、私密文件路径等不应公开的信息。
4. 没有残留模板占位词，例如“待填写”“replace me”“模块一”“模块二”。
5. 文章风格符合用户要求；未确认时默认专业风格。
6. 技术流程、架构或决策类内容优先考虑图表、卡片或分栏表达。
7. 检查是否符合 `docs/ARTICLE_VISUAL_SYSTEM.md`：是否有视觉方案、是否有开头总览块、正文核心视觉块、结尾行动 / 总结块，是否只是套用了固定模板。

## 静态文件

可下载文件放在 `public/files/`，文章使用 `/files/` 开头链接。该目录内容都是公开文件，会随站点发布，不能放私密资料；private 文章不等于 private 文件。

## 发布规则

文章默认公开发布。提交前：

1. 再次检查不应公开的信息和模板占位词。
2. 保持目标文章 `draft: false`。
3. 保持 `private: false`，除非用户明确要求不公开。
4. 运行必要测试和构建。
