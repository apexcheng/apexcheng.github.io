# MDX 文章视觉设计参考

这份文件不是组件 API 文档，而是给后续 Agent 的视觉判断参考。

本博客里的 MDX 可视化，优先服务个人笔记、实践复盘、经验文章和工具方案说明。目标不是把文章做成产品官网，而是让读者更快看见：结论是什么，问题在哪里，数据说明什么，下一步怎么判断。

正式文章写作、重写或可视化改造前，先按 `docs/ARTICLE_VISUAL_SYSTEM.md` 做文章视觉方案；本文件只提供具体风格和写法参考。

## 总原则

1. 先判断文章要表达的结构，再选择视觉形式。
2. 视觉块要帮助读者扫描，不要替代正文思考。
3. 一个视觉块只表达一种关系：概览、指标、对比、路径、时间、案例或结论。
4. 表格只用于规则、字段、轻量参数对比；不要把长解释塞进表格。
5. 流程图只用于分支、依赖、回路或多角色关系；普通顺序不要默认画图。
6. 正式文章可以少用，但要用在关键处；视觉实验文章可以集中展示写法。

## 高视觉密度内容的三种参考风格

这三种风格是写文章时的视觉参考，不是固定组件复用要求。Agent 不要直接把 `AgentFeatureMatrix`、`DefenseArchitectureReport`、`SearchDecisionTree` 硬塞进正式文章；应先判断文章内容结构，再按文章主题重写文案和视觉块。

参考入口：`src/content/posts/visual-lab/screenshot-mdx-replica-demo.mdx`。

当任务涉及文章可视化、复杂排版、视觉实验或“不要只用表格 / Mermaid”时，优先打开这个 Demo 看实际效果，再决定是否参考其中一种风格。

### 1. 功能矩阵风：并列能力 / 模块总览

触发条件：文章里出现“这个系统有哪些能力”“这个方案分成哪些模块”“这个 Agent 能做什么”“这个项目由哪些部分组成”。

视觉骨架：

```text
模块图标
英文或短标签
中文能力标题
1 段短说明
2 到 4 个关键词标签
```

适合：

- Agent 能力矩阵。
- 自动化系统模块地图。
- 博客项目能力总览。
- 商品图 Agent 的 OCR、卖点提炼、主图规划、详情图规划、对话定稿等能力拆解。

不适合：

- 有明显先后顺序的流程。
- 需要是 / 否判断的路径。
- 只有数字和状态的评估结果。

写作动作：

1. 先提炼 4 到 6 个并列模块。
2. 每个模块只写一个核心动作或能力。
3. 每张卡片控制在一个标题、一句说明、几个关键词。
4. 根据文章主题重写文案，不照搬 Demo 里的 OpenClaw 文案。

可以参考的现有文章：

- `src/content/posts/ai-product-image-agent-workflow.mdx`：商品图 Agent 能力矩阵。
- `src/content/posts/blog-project-architecture.mdx`：博客 Agent 驱动能力矩阵。
- `src/content/posts/how-to-use-yingdao-rpa-correctly.mdx`：影刀可覆盖的业务场景矩阵。

### 2. 报告仪表盘风：测试结果 / 验收报告 / 分层保障

触发条件：文章里出现“这个流程稳不稳定”“上线前怎么验收”“有哪些防错层”“风险怎么被挡住”“测试结果如何”。

视觉骨架：

```text
顶部 3 到 5 个指标：评级 / 成功率 / 失败点 / 风险数 / 人工复核数
下方分层列表：L1 / L2 / L3 / L4
每层：层级名 + 解决什么风险 + 右侧状态标签
```

适合：

- 自动化流程稳定性报告。
- Agent 工作流防故障层级。
- 浏览器自动化防误点机制。
- 电商经营看板上线验收。
- 客服售后 RPA 风险控制。

不适合：

- 普通观点表达。
- 没有指标、层级或验收口径的内容。
- 已经用表格能清楚表达的字段清单。

写作动作：

1. 先写顶部指标，指标值必须短，例如 `A-`、`4 层`、`2 处`、`87%`。
2. 再把保障机制拆成 L1 到 L5，不要超过 5 层。
3. 每一层只说明一个风险或防线。
4. 右侧标签用于状态、测试编号、风险等级或适用场景。
5. 根据文章主题重写为“稳定性报告 / 验收标准 / 防错层级”，不要照搬 Demo 里的安全防御文案。

可以参考的现有文章：

- `src/content/posts/automation-agent-workflow.mdx`：Agent 工作流可靠性层级。
- `src/content/posts/ecommerce-dashboard-from-daily-report-to-management-board.mdx`：经营看板验收标准。
- `src/content/posts/ulike-customer-service-rpa-flow-detail.mdx`：客服售后 RPA 风险控制层。

### 3. 深色决策树风：工具选择 / 路径判断 / 是否值得做

触发条件：文章里出现“什么时候用 A，什么时候用 B”“这个需求要不要做”“普通版够不够”“先查哪个维度”“Markdown、MDX、自定义组件怎么选”。

视觉骨架：

```text
起点：用户/业务提出需求
判断 1：是否满足条件 A？
  是 -> 方案 A
  否 -> 判断 2
判断 2：是否满足条件 B？
  是 -> 方案 B
  否 -> 判断 3
判断 3：是否复杂 / 高风险 / 多源？
  是 -> 方案 C
  否 -> 默认方案
侧边卡：适用场景 / 触发信号
底部卡：同步模式 / 异步模式 / 人工兜底
```

适合：

- 工具选型路线。
- 需求是否值得自动化。
- 普通版 / 企业版怎么选。
- RPA / Agent / 人工处理怎么选。
- Markdown / MDX / 自定义组件怎么选。

不适合：

- 只是线性步骤。
- 没有明确判断条件。
- 判断超过 4 层，导致读者走不完。

写作动作：

1. 先确定起点，例如“收到一个新需求”。
2. 只保留 3 到 4 个关键判断问题。
3. 每个判断必须能回答“是 / 否”。
4. 每条路径都要落到明确结果，例如“不做 RPA”“普通版即可”“升级企业版”“先人工跑通”。
5. 侧边卡只放适用场景、触发信号、兜底方式，不放长解释。
6. 根据文章主题重写为专属决策树，不照搬 Demo 里的搜索工具文案。

可以参考的现有文章：

- `src/content/posts/how-to-use-yingdao-rpa-correctly.mdx`：影刀需求是否值得做的决策树。
- `src/content/posts/yingdao-enterprise-rpa-flow-notes.mdx`：普通版 / 企业版影刀选择决策树。
- `src/content/posts/blog-project-architecture.mdx`：Markdown / MDX / 自定义组件选择决策树。
- `src/content/posts/automation-agent-workflow.mdx`：Agent 任务处理路径决策树。

### 使用边界

1. 这三种风格只在文章存在高信息密度结构时使用；普通段落不要强行视觉化。
2. 同一篇正式文章最多选择 1 种作为重点大视觉块，避免视觉块互相抢重点。
3. 如果只是轻量信息，优先用现有 `VisualGrid`、`FeatureCard`、`MetricCard`、`DecisionFlow`。
4. 如果需要接近 Demo 的强视觉效果，可以新建文章专属 Astro 组件；但不要为了单篇普通段落过度组件化。
5. 新建专属组件前先确认它是否明显提升阅读路径；否则用 Markdown 或现有组件。

## Bento 宫格

Bento 宫格适合放文章开头的结构概览，或者把一个经验主题拆成几类重点。

适合：

- 影刀自动化方案的输入、动作、输出、风险。
- 电商经营看板的销售、退款、库存、推广。
- AI 工作流的触发、上下文、工具调用、人工确认。

不适合：

- 字段很多的清单。
- 每项都需要长篇解释的内容。
- 只有两个选项的简单对比。

常用写法：`VisualGrid` 加 `FeatureCard`，每张卡片一个角度。

## 指标卡片

指标卡片适合把数字、比例、状态、耗时、风险等级放出来，让读者先建立量感。

适合：

- 退款率、库存周转天数、滞销 SKU 数。
- 自动化节省时间、失败次数、人工确认节点。
- 文章方案的投入成本、可复用程度、维护风险。

不适合：

- 没有数字或状态的信息。
- 需要解释计算口径的复杂数据。
- 多行文本说明。

常用写法：`VisualGrid` 加 `MetricCard`，`value` 必须短，口径放在正文或 `detail`。

## 对比面板

对比面板适合两个或三个方案的判断，不只是字段对字段。

适合：

- 影刀普通版和企业版的边界。
- 手工日报和经营看板的差异。
- 表格记录、数据库、在线表格三种数据载体选择。

不适合：

- 只是参数清单。
- 每列都只有短词的对比。
- 多于三个方案且每项都很细。

常用写法：两列 `FeatureCard`。如果只是轻量参数，用 Markdown 表格更好。

## 时间线

时间线适合复盘过程，让读者看清事情是怎么演变的。

适合：

- 从手工统计到日报自动化，再到经营看板。
- 一次退款率异常的排查过程。
- 博客文章从素材、草稿、可视化到发布的过程。

不适合：

- 没有先后顺序的模块说明。
- 每一步都有复杂分支的流程。
- 纯任务清单。

常用写法：`DecisionFlow`。如果是日记式复盘，正文小标题也可以。

## 问题链路

问题链路适合展示一个业务问题如何一路传导到结果。

适合：

- 退款率升高：链接、型号、客服备注、库存批次的排查链。
- 库存周转变慢：销量、备货、促销、滞销款的关系。
- AI 工作流失败：输入缺失、上下文错误、工具返回异常、人工兜底。

不适合：

- 单一步骤的操作教程。
- 没有因果关系的列表。
- 只想展示漂亮路径的装饰图。

常用写法：简单链路用 `DecisionFlow`，有分支和回流时用 `Mermaid`。

## 结论卡片

结论卡片适合放文章开头或段落转折处，直接告诉读者这一节的判断。

适合：

- 文章开头的一句话结论。
- 某个方案的推荐结果。
- 风险提醒或发布前检查。

不适合：

- 连续堆多个重点，导致每个都不重点。
- 放普通段落。
- 只做装饰标题。

常用写法：`HighlightBox`。轻量补充用 `Callout`。

## 案例拆解卡

案例拆解卡适合把一个真实或模拟案例拆成背景、动作、结果、复盘。

适合：

- 一个店铺退款率异常的处理。
- 一条影刀流程从失败到稳定的调整。
- 一次 AI 工作流把日报变成看板的实践。

不适合：

- 没有案例，只是概念解释。
- 案例细节很多，需要连续叙述。
- 多个案例横向对比。

常用写法：`FeatureCard` 组合 3 到 4 张卡，或者一张 `HighlightBox` 加列表。

## 决策树

决策树适合帮助读者根据条件选择路径。

适合：

- 要不要用表格、卡片还是流程图。
- 影刀流程要不要上企业版。
- 电商数据异常先查哪个维度。

不适合：

- 没有明确条件的问题。
- 只是线性步骤。
- 判断条件太多，读者看不完。

常用写法：简单判断用 `DecisionFlow`，真正有分支用 `Mermaid`。

## 文章总结横幅

总结横幅适合在文章末尾收束，让读者带走一个可执行判断。

适合：

- “下次遇到同类问题，先做哪三件事”。
- “这套方案适合什么，不适合什么”。
- “保留什么边界，避免什么误用”。

不适合：

- 重复全文所有内容。
- 放太长的操作步骤。
- 引入新观点。

常用写法：`HighlightBox`，内容控制在 1 到 2 段。

## 本博客常见主题的选择建议

| 主题 | 优先视觉形式 | 少用什么 |
| --- | --- | --- |
| 影刀 RPA 流程复盘 | 时间线、问题链路、案例拆解卡 | 大表格 |
| 电商经营看板 | 指标卡片、Bento 宫格、对比面板 | 复杂流程图 |
| 库存周转分析 | 指标卡片、问题链路、决策树 | 纯文字长列表 |
| 退款率排查 | 问题链路、案例拆解卡、指标卡片 | 只有表格 |
| AI 工作流 | 问题链路、决策树、时间线 | 无意义卡片堆叠 |
| 博客项目说明 | Bento 宫格、时间线、结论卡片 | 过大的架构图 |

## 最小选型顺序

```text
先问：这段内容是数字、对比、步骤、因果、案例，还是结论？
数字 -> MetricCard
同类概览 -> VisualGrid + FeatureCard
两个方案取舍 -> 对比面板
线性过程 -> DecisionFlow
因果或分支 -> Mermaid
文章级判断 -> HighlightBox
普通叙述 -> Markdown
```
