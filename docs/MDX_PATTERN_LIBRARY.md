# MDX 文章设计模式库

这份模式库给 Agent 写文章时使用。它解决的不是“组件怎么写”，而是“什么时候不要继续用表格和普通流程图”。

使用前先判断内容结构。能用普通 Markdown 说清楚的，不要强行组件化。

## 1. 开头结论卡

适合什么内容：

- 文章开头的一句话判断。
- 读者先看结论再看原因的经验文章。
- 方案推荐、风险提醒、复盘结论。

不适合什么内容：

- 只是普通引言。
- 作者还没有明确结论。
- 一篇文章连续放多个开头结论卡。

示例用法：

```mdx
<HighlightBox eyebrow="结论" title="退款率先看结构，再看单点异常">
  如果只看总退款率，很容易误判。先拆到店铺、型号、链接和时间段，再决定是商品问题、流量问题还是客服备注问题。
</HighlightBox>
```

常见错误：

- 把整段正文放进结论卡。
- 标题写得像口号，没有具体判断。
- 结论卡后面没有展开原因。

什么时候不用表格：

- 这里不是字段对比，而是文章级判断。

什么时候不用流程图：

- 这里没有步骤和分支，只需要先把判断摆出来。

## 2. Bento 信息宫格

适合什么内容：

- 把一个主题拆成 3 到 4 个观察角度。
- 展示“输入、处理、输出、风险”。
- 展示看板的核心模块。

不适合什么内容：

- 超过 6 个信息块。
- 每个信息块都需要长段解释。
- 严格按时间发生的过程。

示例用法：

```mdx
<VisualGrid columns={4}>
  <FeatureCard title="销售" meta="BOARD" description="看 GMV、订单数和客单价，先确认大盘有没有变。" />
  <FeatureCard title="退款" meta="RISK" description="按链接和型号拆开，避免总退款率掩盖局部异常。" />
  <FeatureCard title="库存" meta="SUPPLY" description="用周转天数判断是否需要补货或清库存。" />
  <FeatureCard title="推广" meta="TRAFFIC" description="把花费、转化和退款放在一起看，而不是只看消耗。" />
</VisualGrid>
```

常见错误：

- 把不同类型的内容混在同一组宫格里。
- 卡片标题过长，读起来像段落。
- 宫格只是装饰，没有帮助读者建立文章地图。

什么时候不用表格：

- 当每一项需要一句解释，且读者更需要先看模块，而不是对齐字段。

什么时候不用流程图：

- 当这些模块是并列关系，不存在先后顺序。

## 3. 指标看板

适合什么内容：

- 有明确数字、比例、等级或状态。
- 想让读者先建立量感。
- 数据看板、复盘结果、自动化收益。

不适合什么内容：

- 只有概念，没有数字。
- 数字口径复杂，需要长说明。
- 每个指标都需要横向比较很多维度。

示例用法：

```mdx
<VisualGrid columns={3}>
  <MetricCard label="退款率" value="6.8%" detail="比上周高 1.4 个百分点，先查链接和型号。" />
  <MetricCard label="周转天数" value="42 天" detail="超过目标区间，优先看滞销库存。" tone="cyan" />
  <MetricCard label="人工复核" value="2 处" detail="价格异常和库存缺口仍需人工确认。" tone="green" />
</VisualGrid>
```

常见错误：

- `value` 写完整句子。
- 没有说明指标口径。
- 数字没有业务含义，只是为了填卡片。

什么时候不用表格：

- 当指标少于 4 个，而且每个指标都需要被突出扫描。

什么时候不用流程图：

- 指标是状态，不是过程。

## 4. 左右对比面板

适合什么内容：

- 两个方案、两种状态、两种做法之间的取舍。
- 每个方案都有 2 到 4 条解释。
- 想让读者看完后能做选择。

不适合什么内容：

- 只是参数字段。
- 超过三个方案。
- 对比项很多且需要严格逐行对齐。

示例用法：

```mdx
<VisualGrid columns={2}>
  <FeatureCard title="继续手工日报" meta="旧做法" description="适合量小、变化慢、只需要记录结果的阶段。">
    <ul>
      <li>维护成本低</li>
      <li>难以及时发现异常</li>
    </ul>
  </FeatureCard>
  <FeatureCard title="升级经营看板" meta="新做法" description="适合多店铺、多型号、需要持续追踪异常的阶段。">
    <ul>
      <li>能把退款、库存和推广放在一起看</li>
      <li>需要先统一字段口径</li>
    </ul>
  </FeatureCard>
</VisualGrid>
```

常见错误：

- 把所有差异都塞进去，导致卡片变成长文。
- 两边信息粒度不一致。
- 只有好坏判断，没有适用边界。

什么时候不用表格：

- 当每个方案需要讲边界、取舍和结论，而不是逐字段对齐。

什么时候不用流程图：

- 当重点是横向比较，不是执行顺序。

## 5. 问题链路图

适合什么内容：

- 一个问题从现象到原因再到动作的传导关系。
- 业务异常排查。
- AI 工作流失败链路。

不适合什么内容：

- 单纯步骤教程。
- 没有因果关系的列表。
- 图会比正文更难读的复杂场景。

示例用法：

```mdx
<Mermaid
  title="退款率异常排查链路"
  caption="先缩小范围，再判断是商品、流量还是服务问题。"
  chart={`flowchart LR
    A[总退款率升高] --> B[拆店铺]
    B --> C[拆链接]
    C --> D[拆型号]
    D --> E{异常集中吗}
    E -->|是| F[查商品和批次]
    E -->|否| G[查流量和客服备注]`}
/>
```

常见错误：

- 把所有排查点都画进一张图。
- 节点文字太长。
- 明明是线性步骤，却画成复杂流程图。

什么时候不用表格：

- 当重点是因果传导，不是字段罗列。

什么时候不用普通流程图：

- 如果只是 3 到 5 步线性过程，用 `DecisionFlow` 更轻。

## 6. 时间线

适合什么内容：

- 项目复盘。
- 自动化方案逐步演进。
- 从问题发现到验证落地的过程。

不适合什么内容：

- 并列模块介绍。
- 需要复杂条件判断。
- 没有明显时间顺序。

示例用法：

```mdx
<DecisionFlow
  title="日报升级为经营看板的过程"
  items={[
    {
      label: '阶段 1',
      title: '先统一字段',
      description: '把店铺、链接、型号、日期和售后原因整理成稳定口径。',
    },
    {
      label: '阶段 2',
      title: '再自动采集',
      description: '用影刀减少重复下载和复制，保留异常处人工确认。',
    },
    {
      label: '阶段 3',
      title: '最后做看板',
      description: '把退款率、库存周转和推广消耗放进同一个判断页面。',
    },
  ]}
/>
```

常见错误：

- 把时间线写成任务清单。
- 每一步描述太抽象。
- 时间线里突然插入并列对比。

什么时候不用表格：

- 当读者需要理解演进顺序，不是对齐字段。

什么时候不用流程图：

- 当没有分支、回路和条件判断时，不需要 Mermaid。

## 7. 案例拆解卡

适合什么内容：

- 一个具体案例的背景、动作、结果、复盘。
- 文章中段用来把抽象方法落到场景。
- 电商、影刀、AI 工作流的轻量案例。

不适合什么内容：

- 案例细节需要长篇叙述。
- 多个案例需要严谨比较。
- 还没有明确动作和结果。

示例用法：

```mdx
<VisualGrid columns={2}>
  <FeatureCard title="背景" meta="CASE" description="某店铺保护套退款率突然升高，总退款率只提示了异常。" />
  <FeatureCard title="动作" meta="ACTION" description="按链接、型号和客服备注拆开，发现问题集中在一个新批次。" />
  <FeatureCard title="结果" meta="RESULT" description="暂停推广该链接，先处理库存和详情页说明。" />
  <FeatureCard title="复盘" meta="REVIEW" description="看板需要同时保留总览和下钻维度，否则只能看到滞后的结果。" />
</VisualGrid>
```

常见错误：

- 没有真实动作，只写观点。
- 结果没有和动作对应。
- 卡片过多，案例被拆碎。

什么时候不用表格：

- 当案例是叙事结构，不是字段记录。

什么时候不用流程图：

- 当案例只是拆解四个侧面，不需要画路径。

## 8. 决策树

适合什么内容：

- 帮读者根据条件选择下一步。
- 文章写法选择。
- 自动化方案是否升级。

不适合什么内容：

- 没有明确判断条件。
- 只是说明概念。
- 分支超过 6 个。

示例用法：

```mdx
<Mermaid
  title="文章视觉模式选择"
  caption="先看内容结构，再决定要不要用图。"
  chart={`flowchart TD
    A{这段内容是什么}
    A -->|数字状态| B[指标卡片]
    A -->|两种方案| C[对比面板]
    A -->|线性过程| D[时间线]
    A -->|因果分支| E[问题链路或决策树]
    A -->|普通叙述| F[Markdown 段落]`}
/>
```

常见错误：

- 条件写得含糊，读者无法判断。
- 每个分支又继续展开太多层。
- 只为了替代目录而画树。

什么时候不用表格：

- 当读者需要沿条件走路径，不是查找某个字段。

什么时候不用流程图：

- 如果只有一个判断和两个结果，可以直接用正文或对比面板。

## 9. 文章总结横幅

适合什么内容：

- 文章末尾收束。
- 给出下次可复用的判断。
- 强化边界和行动建议。

不适合什么内容：

- 引入新观点。
- 重复全文大纲。
- 放长清单。

示例用法：

```mdx
<HighlightBox eyebrow="收束" title="视觉设计的目标是让判断更快出现">
  这类文章不需要把每段都做成组件。最好的状态是：读者先通过卡片和链路看懂骨架，再回到正文理解细节。
</HighlightBox>
```

常见错误：

- 总结横幅写得比正文还长。
- 没有给出可带走的判断。
- 和开头结论重复但没有推进。

什么时候不用表格：

- 总结不是数据结构。

什么时候不用流程图：

- 总结是收束，不是新流程。

## 10. 检查清单块

适合什么内容：

- 发布前检查。
- 自动化流程上线前确认。
- 数据看板字段口径确认。

不适合什么内容：

- 需要解释背景的复杂问题。
- 有分支判断的选择题。
- 太长的操作手册。

示例用法：

```mdx
<HighlightBox eyebrow="检查" title="看板发布前先确认三件事">
  <ul>
    <li>字段口径是否和日报一致。</li>
    <li>异常指标是否能下钻到店铺、链接和型号。</li>
    <li>需要人工确认的节点是否被明确标出来。</li>
  </ul>
</HighlightBox>
```

常见错误：

- 把操作步骤写成检查清单。
- 清单项互相重叠。
- 没有可验证动作。

什么时候不用表格：

- 当重点是逐项确认，不需要横向比较。

什么时候不用流程图：

- 当清单没有顺序依赖。

## 选择口诀

```text
能一句话讲清 -> Markdown
需要强调判断 -> HighlightBox
有数字状态 -> MetricCard
有并列模块 -> VisualGrid + FeatureCard
两个方案取舍 -> 对比面板
有先后顺序 -> DecisionFlow
有因果分支 -> Mermaid
有案例结构 -> 案例拆解卡
```
